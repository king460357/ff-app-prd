# AI_GUIDE — 协作 AI 阅读 PRD 通用规范

> 本文件给所有协作 AI（前端 / 后端 / 数据库 / QA / 其他 PM）使用——拉到本仓库后**第二份必读**（第一份是 README.md）。
>
> 本规范是"读 PRD 时该理解的章节意义 + 视觉规范继承 + 文件命名约定"，不是产品经理写 PRD 的内部流程。

---

## 1. PRD 双产物（2026-05-10 起）

每份 PRD 是 `.md` + `.html` 双产物：

| 产物 | 受众 | 用途 |
|---|---|---|
| `{date}_{需求}_PRD.md` | 研发 AI / 决策 AI / sync 工具 | **真理源**——事实层始终以 markdown 为准 |
| `{date}_{需求}_PRD.html` | 产品经理自审 / 老板评审 / 研发偶尔看全貌 | **派生展示版**——含 dark theme + 实物 UI 渲染 + 顶部"怎么打开"段 + 简单交互演示 |

**冲突时**：`.md` 优先于 `.html`（HTML 严格忠于 markdown，事实分叉视为 bug）。

## 2. PRD 主文档章节结构（10-12 章固定模板）

每份 PRD 主文档 `{YYYYMMDD}/PRD/{需求}/{日期}_*_PRD.md` 都按以下章节组织：

| 章节 | 内容 | 主要服务角色 |
|---|---|---|
| frontmatter `ai_context` | 5 类角色路由（每个角色应该读哪几章）| 全部角色，**进 PRD 第一眼读** |
| 顶部 banner | 真理源约定（`.md` 优先于 `.html`）| 全部角色 |
| 1 产品概述 | 背景 + 问题陈述 + 目标 + 优先级与排期 | PM / 全部 |
| 2 目标用户与使用场景 | ≥3 个具体场景（不是空泛画像）| PM / 产品 / UI |
| 3 核心用户动线 | Mermaid 流程图 | 全部 |
| 4 全局视觉与界面总览 | 视觉系统 + 整体一图速览 + 模块导航 + 不做什么 | UI / 前端 |
| 4.1.1 组件库限定 | shadcn/ui 硬约束 + ABSOLUTE REQUIREMENTS（UI 提示词头部）| UI / 前端 |
| 5 功能模块详述 | 每个 5.x 模块自包含：UI / 交互 / 状态 / 数据 / 验收 | 全部（按"目标读者"标签）|
| 6 文案规范 | 双受众分开（终端用户文案 vs 内部术语）| 前端 / UI / 运营 |
| 7 非功能性需求 | 性能 / 兼容 / 安全 / 可观测 | 后端 / QA |
| 8 影响范围矩阵 | 8.1-8.7 七个子节，给研发拆任务 | **研发第一阅读章节** |
|   8.3.1 错误码全集 | 后端必读 | 后端 |
|   8.4.1 完整 DDL | 数据库必读 | 数据库 |
| 9 待确认问题 | 含默认假设 | 全部 |
| 10-12 上线指标 / 红线 / 工时 | 收尾 | 全部 |

## 3. 5.x 模块的 12 项必填（"自包含"是核心）

每个 🔴 核心模块都按以下 12 项展开，确保**读完一节就掌握一个界面块的全部信息**——不需要跳转其他章节：

| # | 项目 | 是什么 |
|---|---|---|
| 1 | 功能说明 | 解决什么问题、面向谁、用于什么场景 |
| 2 | 触发条件 | 什么时候启动 |
| 3 | 入口位置 | 用户从哪里能进入 |
| 4 | 页面结构 | 整体布局描述 |
| 5 | UI 示意 | ASCII 线框图 + UI 提示词（HTML 派生产物含实物 UI 渲染）|
| 6 | 按钮交互 | 每个按钮 6 项必填（默认是否可点击 / 是否弹窗 / 状态变化 / 是否可重复 / 成功后去哪 / 失败提示）|
| 7 | 弹窗规则 | 每个弹窗 5 项必填（标题 / 正文 / 按钮文案 / 取消结果 / 确认结果）|
| 8 | 状态清单 | 空 / 加载中 / 成功 / 失败 / 部分失败 / 置灰 6 类全覆盖 |
| 9 | 异常处理 | 空内容 / 超长 / 网络异常 / 无权限 / 并发 / 格式不符 6 类边界 |
| 10 | 联动规则 | 与其他模块的同步关系 |
| 11 | 数据规范 | 字段名 / 类型 / 长度 / 必填 / 默认值 / 校验规则 + 接口契约 |
| 12 | 验收标准 | 可观察、可判定（不允许"流畅、美观"等模糊词）|

## 4. 视觉规范（dark theme + shadcn/ui）

### 4.1 主题三锁（必须遵守）

线上唯一主题是 **Dark**——三锁保证不被"误认为有 Light 模式"：

1. `globals.css :root` 直接是 dark（无 `.dark` 类切换）
2. `layout.tsx` `themeColor: '#111111'`
3. `layout.tsx` `<Toaster theme="dark">`

色值唯一事实源：**autoflow `frontend/app/globals.css:6-45`**。任何 PRD 表格如与 globals.css 冲突，以 globals.css 为准。

### 4.2 核心色值（dark theme）

| 用途 | HEX | CSS 变量 |
|---|---|---|
| 页面背景 | `#111111` | `--background` |
| 卡片 / 弹窗 / Popover 背景 | `#1F2937` | `--card` / `--popover` |
| Sidebar 背景 | `#1a1a1a` | `--sidebar` |
| 主文字 | `#F9FAFB` | `--foreground` |
| 次文字（描述 / 来源）| `#9CA3AF` | `--muted-foreground` |
| 主色（CTA / 链接 / 焦点环）| `#3B82F6` | `--primary` / `--ring` |
| 边框 / 分隔线 | `#374151` | `--border` |
| 危险（删除 / 失败）| `#EF4444` | `--destructive` |
| 三档评分（成功率 / 失败热度图）| 🟢 `#10B981` / 🟡 `#F59E0B` / 🔴 `#EF4444` | `--score-green/yellow/red` |

### 4.3 控件密度（shadcn/ui）

参照 autoflow `frontend/components/admin/stats/user-detail-drawer.tsx` 的 `ClusterRow` 密度：

- **Tabs**：细字标签 + 1-2px 下划线激活（**不是**填色 pill 按钮）
- **Cards**：`<CardContent className="p-3">` + 1px `border-border` + `shadow-sm`（**不是**大圆角 + 强阴影 + 24px+ 内边距）
- **Buttons**：`size="sm"` ~28px 高（**不是** oversized 40px+ 大尺寸亮蓝 pill）
- **Tables**：细 1px 横线 + `text-sm` + 无斑马纹
- **Forms**：`<FormField>` + `<FormDescription className="text-xs text-muted-foreground">` 显示来源行
- **图标**：lucide-react inline `<Search h-4>` / `<Calendar h-4>` 等（**仅用于控件 affordance**）

### 4.4 品牌 emoji 强制清单（绝不替换为 lucide 图标，绝不剥）

不同需求有不同的品牌 emoji，**研发实施时必须保留**——这些是产品语义的一部分：

- ✅ / 🟡 / 🔴 / ⚠️ / 📭：状态色点（核验状态 / 成功率档位 / 报白状态等）
- 📅：日期前缀
- 💰：高成本信号
- 📏：编辑距离异常信号
- 🤖：AI cron / AI 自动入库
- ⏪：快照回滚
- ⚙️：设置开关
- ⬆️ / ⬇️：导入 / 导出
- 📋：变更记录
- 📊：通过率统计

具体每份 PRD 的 emoji 强制清单见 4.1.1.A 节。

### 4.5 全文案中文（绝不缩写）

状态徽章 / 按钮文案必须显示完整中文短语：

- "✅ 已核验" 而非 "已"
- "🟢 高成功率 / 🟡 中成功率 / 🔴 低成功率" 而非 "高/中/低"
- "[立即替换] / [保持不变] / [按净化版提交（推荐）]" 不缩写为 "替换 / 不变 / 净化"

## 5. 文件命名约定

### 5.1 PRD 双产物

```
{YYYYMMDD}/PRD/{需求名}/
├── {YYYYMMDD}_{需求名}_PRD.md     # 真理源（AI 读）
└── {YYYYMMDD}_{需求名}_PRD.html   # 派生展示版（人类读）
```

主目录里**只允许存在一份"当前版"双产物**——出现 `xxx_PRD_v2.md` 这种是错误的，应把旧版移入 `_archive/` 并在文件名末尾加日期。

### 5.2 决策上下文

```
{YYYYMMDD}/decision-making/
├── {需求名}/{YYYYMMDD}_{需求}.md   # 单需求决策表
└── _汇总/{YYYYMMDD}_需求决策汇总.md   # 批次汇总
```

## 6. 真理源优先级（冲突时如何取舍）

按优先级**自顶向下**：

1. **autoflow 主线代码现状**（`frontend/app/globals.css:6-45` 等具体代码）—— 实际事实
2. **PRD `.md` 主文档**（本仓库 `{YYYYMMDD}/PRD/`）—— 设计真理源
3. **决策上下文文档**（本仓库 `{YYYYMMDD}/decision-making/`）—— 设计依据
4. **PRD `.html` 派生展示版** —— 严格忠于 `.md`，仅供人类查看

研发拿到任务时：**先核对 `.md` 是否与代码现状一致 → 按 `.md` 实现 → `.html` 仅作辅助理解**。

## 7. 跨需求 / 跨版本协调

### 7.1 同发布周期内的需求

如果两份需求在同一发布周期共用同一对象（同一 UI / 同一表 / 同一接口），**必须在 PRD 内互相 cross-reference**——通常一份当主权威源，另一份引用。

### 7.2 跨版本叠加

如果后版本要在前版本基础上**追加**功能（不是新建独立组件），后版本 PRD 必须明确：
- 前版本已上线的部分**全部保留**（不武断砍掉）
- 本版本仅追加新代码块到同一文件，不重写

研发实施时严格按上线顺序：先做早版本 → 再叠加后版本，每次都保留前版本能力。

---

## 8. 反馈与修订

发现 PRD 内容不清晰、与代码现状冲突、或需要补充约束 —— **通过常规协作渠道反馈给产品总监**，不要在本仓库直接改 PRD。产品总监会按以下流程修订：

1. 在 product 主仓更新 PRD `.md`（→ AI 同步重生 `.html`）
2. 同步到本仓库（`bash scripts/sync-to-prd-repo.sh --batch YYYYMMDD`）
3. PRD frontmatter `version` 字段递增 + `changelog` 加一行说明改动

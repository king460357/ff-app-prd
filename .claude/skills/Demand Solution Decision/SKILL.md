# 需求决策表生成器（Decision Table Generator）

## 核心定位

这个 skill 帮产品经理把"一句模糊的用户反馈"通过**三轮对话逐步确认**，最终生成一份**可以直接拍板、直接提交给 prd-writer 的最终决策方案**。

**与 prd-writer 的关系**：
- `decision-table-generator` 在前：决定"要不要做、怎么做、做什么范围"
- `prd-writer` 在后：拿到最终决策方案后，展开完整 PRD 交付开发
- 整条链路：**用户反馈 → 决策方案 → 完整 PRD → 开发**

### 与 PRD 之间的强制门禁（本仓库约定）

**默认**：在用户对「决策表 / 推荐方案 / 范围与优先级」给出**显性确认**之前，**不得**启动 `prd-writer-master` 撰写完整 PRD，也不得把主 PRD 写入 `docs/{YYYYMMDD}/PRD/...`。

- **AI 行为**：输出决策草案后，用**简短确认清单**收尾，并**明确提问**（例如：是否同意推荐方案？需改哪几条？）。**停下等待用户回复**。  
- **用户确认**示例：「按推荐方案写 PRD」「第 3 条改为不做，其余确认」「跳过确认直接写」（最后一项视为授权跳过门禁）。  
- **目的**：避免未对齐意图即产出大文档导致返工。

### 用户分群与策略矩阵（交付 prd-writer 前的必选项）

凡需求涉及 **不同人群不同策略**（商业化、门闸、广告形态、权益、实验桶等），决策表输出中**必须**包含一节 **「用户分群真值表」**（或等价表格），且：

1. **主目标人群**：用 PM 可理解 + 研发可实现的描述（付费史、活跃、行为标签等）；禁止仅写「未付费用户」等笼统词。  
2. **每一类非主人群**：写明是否享受本需求、若享受是否与主人群一致；不一致则写**单独策略**或写「待 PM 确认」**不得**默认与主人群相同。  
3. 若 PM 未回复非主人群：**决策表须标为未完成**，并建议**不要**进入 prd-writer 终稿；除非 PM 明确说「主人群已定、其余后补」并在表中列出 **⚠️ 后补清单**。

与 **「与 PRD 之间的强制门禁」** 叠加：先决策（含分群表）→ PM 确认 → 再 PRD。

## 第零步：识别场景，决定启动方式

收到用户输入时，先判断属于哪种场景：

| 用户的输入特征 | 场景 | 处理方式 |
|---------------|------|---------|
| 给了完整的「用户反馈+问题摘要+反馈ID」 | A. 完整输入 | 直接进入第一步分析 |
| 只给了用户反馈，没给摘要和ID | B. 半完整输入 | AI 自己提炼问题摘要并标注 `[AI 推断]`，反馈ID 写"待补充"，直接分析 |
| 只给了一句话或一团乱的聊天记录 | C. 信息很少 | 先帮用户提炼出"用户反馈"和"问题摘要"，标 `[AI 推断]`，然后分析 |
| 多条反馈（2条及以上） | D. 批量场景 | 先输出优先级总览，等确认后逐条进入三轮决策 |
| 完全没说反馈、只说"我想做个 xx 功能" | E. 不适用 | 提示用户：本 skill 用于把已有反馈转化为决策；从零想新功能请用 `prd-writer` |

⚠️ **信息不全不要反问太多**——能推断的就推断+标注 `[AI 推断]`，让用户拿到结果再调整。只有反馈完全无法理解时才反问。

## 第一步：内部分析（不输出，纯思考）

### 1.1 提炼问题本质
- 用户真正在抱怨什么？功能缺失、体验问题、还是认知偏差？
- 是个例还是高频？（没说则标"频次待确认"）
- 影响核心流程、付费流程、还是边缘功能？

### 1.2 判断优先级

| 优先级 | 判断标准 |
|--------|---------|
| **P0** | 阻塞核心流程 / 影响付费 / 数据安全 |
| **P1** | 高频痛点 / 影响留存 |
| **P2** | 体验优化 / 中频问题 |
| **P3** | 锦上添花 / 低频长尾 |

判断不出来时：给最可能的优先级 + 标注"待确认（依赖：xx 数据）"。

### 1.3 判断产品决策

| 决策 | 适用场景 |
|------|---------|
| **处理** | 问题真实 + 优先级足够 + 方案可行 |
| **暂缓** | 问题真实但当前不是最优解 / 缺数据 / 排期冲突 |
| **拒绝** | 与产品定位冲突 / 伪需求 / ROI 过低 |
| **合并** | 与已有需求/规划功能重叠 |

### 1.4 判断方案数量
- **小需求**（改文案、加 tooltip 等）：1 个方案
- **中等需求**（交互调整、新增功能）：A 快速版 + B 完整版
- **大需求**（流程改造、新模块）：A 快速版 + B 完整版 + C 长期版

---

## 第一点五步：批量反馈时先输出优先级总览（单条跳过）

收到 2 条及以上反馈时，完成内部分析后，先输出总览：

```
共收到 N 条反馈，整理如下：

| # | 需求标题 | 优先级 | 建议决策 | 一句话说明 |
|---|---------|--------|---------|-----------|
| 1 | <标题>  | P0     | 处理    | <一句话> |
| 2 | <标题>  | P1     | 处理    | <一句话> |
| 3 | <标题>  | P2     | 暂缓    | <一句话> |

从第 1 条开始逐条决策。准备好了吗？
```

⚠️ 输出总览后停下来，等用户确认后才开始第一条的三轮决策。

---

## 第二步：三轮对话式决策

**每条需求严格按以下三轮进行，每轮等用户回复后才进入下一轮。绝不跳轮。**

批量场景下，每条开始前标注进度：
> **「需求 2 / 4」** — 视频列表卡顿优化（P1）

---

### 第一轮：问题确认 + 优先级

输出格式：

```
📋 **第一轮：问题确认**

**问题本质**
[用 1-2 句话说清楚：用户在抱怨什么、影响范围是什么]

**建议优先级：[P0/P1/P2/P3]**
理由：[对照判断标准说明为什么]
影响范围：[影响核心流程/付费/留存/体验，哪个]

[如有 AI 推断内容，明确标出]

---
请确认（直接回复数字即可）：
① 分析准确，继续选方案
② 问题描述需要调整 → 告诉我哪里不对
③ 优先级不对 → 告诉我应该是几
④ 这条决策：暂缓 / 拒绝 / 合并 → 直接跳到生成最终方案
```

⚠️ 收到用户回复后：
- ①→ 进入第二轮
- ②③→ 修订后重新输出第一轮，再等确认
- ④→ 跳过第二三轮，直接进入第三步生成该条的最终方案（决策写"暂缓/拒绝/合并"）

---

### 第二轮：方案选择

第一轮确认后输出，格式：

```
✅ 问题确认完毕。

🔀 **第二轮：选择解决方案**

| | 方案A [名称] | 方案B [名称] | 方案C [名称，大需求才有] |
|---|---|---|---|
| **做什么** | [1-2条] | [1-2条] | [1-2条] |
| **用户体验变化** | [描述] | [描述] | [描述] |
| **研发成本** | 低 | 中 | 高 |
| **QA复杂度** | 低 | 中 | 高 |
| **主要风险** | [1条] | [1条] | [1条] |
| **是否推荐** | [是/否] | [✓ 推荐] | [视情况] |

---
请选择（直接回复即可）：
① 选方案A  ② 选方案B  ③ 选方案C（如有）
④ 方案都不合适，我来描述我的想法
```

⚠️ 收到用户回复后：
- 选了某方案→ 记录选定方案，进入第三轮
- ④→ 用户描述后，AI 重新整理一个新方案，回到第二轮再确认

---

### 第三轮：范围确认

第二轮确认后输出，格式：

```
✅ 方案[X] 已选定。

📌 **第三轮：确认本次范围**

**做：**
- [实现项 1]
- [实现项 2]
- [实现项 3]

**不做（本次明确排除）：**
- [排除项 1]
- [排除项 2]

**用户流程：**
入口：[用户从哪里进入]
操作：[做什么]
期望结果：[得到什么]

**验收标准：**
- [可验证标准 1]
- [可验证标准 2]
不能上线：[红线 1] / [红线 2]

---
请确认（直接回复即可）：
① 范围没问题，生成最终决策方案
② 需要调整范围 → 告诉我哪里改
③ 需要调整验收标准 → 告诉我
```

⚠️ 收到"①"后，立即进入第三步，生成该条需求的最终决策方案。

---

## 第三步：生成最终决策方案

三轮全部确认后，输出该需求的最终决策方案。格式设计为**可直接复制提交给 prd-writer**，干净无冗余。

**输出完文档内容后，立即将其保存为 Markdown 文件到项目内对应目录：**
- 文件路径：`docs/decision-making/{项目名}/{需求名称}/{YYYYMMDD}_{需求简称}.md`
- 项目名 / 需求名称从用户输入或上下文推断；不存在则用 Bash `mkdir -p` 创建
- 需求简称：从需求标题中提取 2-4 个核心词（如"任务队列健壮性"、"报白同步延迟"），不超过 10 个字
- ⚠️ **禁止写入 `~/Desktop/` 或任何项目目录之外的位置**
- 保存完成后告知用户：「已保存到项目内：`docs/decision-making/{项目名}/{需求名称}/{文件名}.md`」

```markdown
# [需求简称]

> **完整需求名**：[需求标题全称]
> 决策时间：[日期] | 优先级：[P0/P1/P2/P3] | 决策：[处理/暂缓/拒绝/合并] | 选定方案：[A/B/C - 方案名称]

---

## 需求背景
[2-4 句话描述问题来源、用户反馈、影响范围]

## 选定方案：[方案名称]
[1-2 句话描述这个方案的核心思路]

## 本次做什么
- [实现项 1]
- [实现项 2]
- [实现项 3]

## 本次不做什么
- [排除项 1]
- [排除项 2]

## 用户流程
- **入口**：[用户从哪里进入]
- **操作**：[做什么]
- **期望结果**：[得到什么]

## 页面与交互
- 涉及页面：[页面名称]
- 涉及元素：[按钮/弹窗/状态等]
- 交互变化：[前后差异]

## 验收标准
- [可验证标准 1]
- [可验证标准 2]
- **不能上线的情况**：[红线]

## 上线验证指标
- **指标**：[指标名]（[定义]）
- **目标值**：[数值 或 "待产品确认"]
- **数据来源**：[埋点/日志/看板]
- **是否需要埋点**：[是/否，原因]

---
✅ **此方案已确认。**
提交给 prd-writer 时，建议附带此文档并说：
「基于以下最终决策方案，帮我展开完整 PRD」
```

**决策为暂缓/拒绝/合并时的简化格式：**

```markdown
# [需求简称]

> **完整需求名**：[需求标题全称]
> 决策时间：[日期] | 优先级：[P0/P1/P2/P3] | 决策：[暂缓/拒绝/合并]

## 决策理由
[说明为什么暂缓/拒绝/合并，1-3句]

## 后续建议
- [暂缓→说明什么条件下可以重新评估]
- [合并→说明合并到哪个需求/功能]
- [拒绝→说明是否有替代方案]

---
⏸️ **此需求暂不进入 PRD 流程。**
```

## 第四步：批量场景——所有需求决策完毕后输出汇总文档

所有条目的三轮决策全部完成后，自动输出一份汇总，并保存到项目内：
- 文件路径：`docs/decision-making/{项目名}/_汇总/{YYYYMMDD}_需求决策汇总.md`
- 不存在则用 Bash `mkdir -p` 创建 `_汇总/` 子目录
- ⚠️ **禁止写入 `~/Desktop/`**
- 保存完成后告知用户：「已保存到项目内：`docs/decision-making/{项目名}/_汇总/{文件名}.md`」

```markdown
# [产品名] 需求决策汇总

> 决策日期：[日期] | 共 N 条需求 | 处理：X 条 | 暂缓：X 条 | 拒绝：X 条

---

## 待开发需求（按优先级排序）

### 1. [需求简称]（P0）
> **完整需求名**：[需求标题全称]

[粘贴该需求的完整最终决策方案正文]

---

### 2. [需求简称]（P1）
> **完整需求名**：[需求标题全称]

[粘贴该需求的完整最终决策方案正文]

---

## 暂缓 / 拒绝需求

| 需求简称 | 完整需求名 | 决策 | 理由 | 后续建议 |
|---------|---------|------|------|---------|
| [简称C] | [全称] | 暂缓 | [理由] | [条件] |
| [简称D] | [全称] | 拒绝 | [理由] | — |

---
✅ **以上为本次决策全部内容。**
「处理」类需求可按优先级顺序逐条提交给 prd-writer 生成 PRD。
```

## 通用原则（红线）

1. ❌ **不要扩展用户未提出的功能**——反馈说"搜索慢"，只评估搜索，不顺手提"加筛选器"
2. ❌ **不要生成接口、数据库、架构方案**——这是研发的事
3. ❌ **不要编造数据**——优先级判断缺数据时，写"待确认（依赖：xx 数据）"
4. ✅ **AI 推断必须标注 `[AI 推断]`**
5. ✅ **严格等用户每轮回复后再进入下一轮**——不跳轮，不连续输出
6. ✅ **每个字段都要有内容**——判断不了的写"待确认"，不留空

## 完整示例（新流程）

**用户输入**：
> "App 里搜索商品老是搜不到我想要的，搜'红色连衣裙'结果一堆裙子都不是红色的，太烦了。"

**内部判断**：P1，处理，中等需求→出 A+B 两方案

**第一轮输出** → 用户回复"①"  
**第二轮输出**（方案对比表）→ 用户回复"②选方案B"  
**第三轮输出**（范围确认）→ 用户回复"①没问题"  
**最终决策方案输出** → 用户直接复制提交给 prd-writer

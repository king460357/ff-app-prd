# open-questions-triage Skill

把当期需求目录的 `OPEN_QUESTIONS.md` 转化为可批量回答的分诊报告，让产品总监高效答疑。

## 一、何时触发本 skill

用户出现以下任一情况时使用：

1. 说"看下 OPEN_QUESTIONS" / "今天的问题"
2. 在多个项目工作，要批量看所有项目的 Q&A
3. 想知道哪些问题"重复出现"该升级 FAQ
4. 想查 OPEN_QUESTIONS 阻塞情况

## 二、本 skill 的核心步骤

### 步骤 1：定位 OPEN_QUESTIONS.md 文件

| 场景 | 路径 |
|---|---|
| 当前批次某个需求 | `docs/{YYYYMMDD}/PRD/{需求}/OPEN_QUESTIONS.md` |
| 单一批次所有需求 | `docs/{YYYYMMDD}/PRD/*/OPEN_QUESTIONS.md`（用 Glob 扫描） |
| 全部批次 | `docs/*/PRD/*/OPEN_QUESTIONS.md` |

如用户没指定，默认扫"全部批次所有需求"。

### 步骤 2：读取并解析每条 Q

每条问题按以下标准格式解析（来自主 PRD 决策上下文章节）：

```markdown
## Q[编号] | [角色] | [模块] | [日期]
**类型**：歧义 / 缺漏 / 跨角色冲突 / 业务判断 / 优先级
**问题**：...
**我倾向方案**：...
**影响**：...
**当前状态**：未裁定 / 用户已裁定（[日期]：[结论]）
```

提取字段：编号 / 角色 / 模块 / 日期 / 类型 / 问题 / 倾向方案 / 影响 / 状态 / 是否阻塞（顶部有 🔴 阻塞 标记）。

### 步骤 3：分类 + 排序

按以下优先级排序（顶部最紧急）：

```
1. 🔴 阻塞条目（无论类型）
2. 业务判断 / 优先级争议（必须用户决定）
3. 跨角色冲突（影响多人）
4. 歧义（一句话两种理解，影响实现）
5. 缺漏（PRD 没覆盖的边界）
6. 已裁定的（仅汇总数量，不展开）
```

### 步骤 4：基于 PRD / 决策上下文起草回答候选（关键步骤）

每条**未裁定问题**，本 skill 要主动读以下文件给出**回答候选**：

| 读取文件 | 用途 |
|---|---|
| 当期主 PRD（`docs/{YYYYMMDD}/PRD/{需求}/{YYYYMMDD}_{需求}_PRD.md`）| 找到相关章节看是否已覆盖（含第 1 章决策上下文 / 第 5 章模块详细 / 第 8 章影响范围）|
| 主 PRD 第 1 章「考虑过但放弃的设计」段 | 看是否已被放弃 |
| 主 PRD 第 1 章 FAQ 段 | 看是否已有标准答案 |
| 主 PRD 第 1 章硬约束段 | 看是否违反硬约束 |
| `docs/confirmed-decisions.md` | 看是否已有相关历史决策 |

回答候选的判断逻辑：

| 情况 | 回答候选模板 |
|---|---|
| PRD 中已明确（开发端漏看）| "PRD `{章节}` 已说明：`{原文摘录}`。请按 PRD 执行。" |
| 决策上下文已废弃 | "v{版本} 已放弃此方案，原因见主 PRD 第 1 章「考虑过但放弃的设计」段。请按当前版本执行。" |
| 命中硬约束 | "违反硬约束 {N}：`{约束内容}`。建议按倾向方案修订实现，避开硬约束；或升级为决策表层重新评估硬约束。" |
| 已有相关历史决策 | "见 `confirmed-decisions.md` 决策记录 #{N}：`{结论摘要}`。" |
| **PRD 真未覆盖**，需用户判断 | 给出 2-3 个备选方案 + 各自影响分析，标注 "[待用户确认]" |

**回答候选不是终答**——只是省去用户思考的草稿。用户看到草稿后确认 / 修改 / 否决。

### 步骤 5：识别 FAQ 候选

扫描所有问题（含已裁定的），找出"重复出现 ≥ 3 次"的同类问题。判断标准：

- 类型 + 模块相同
- 或问题语义相似（不要求字面一致）

输出"FAQ 候选清单"，建议产品总监搬到主 PRD 第 1 章 FAQ 段固化。

### 步骤 6：输出分诊报告

```markdown
# OPEN_QUESTIONS 分诊报告

**扫描范围**：docs/{YYYYMMDD}/PRD/{需求名称}/OPEN_QUESTIONS.md
**扫描时间**：2026-05-XX HH:MM
**总问题数**：N
- 🔴 阻塞：A 条（必看）
- 业务判断：B 条
- 跨角色冲突：C 条
- 歧义：D 条
- 缺漏：E 条
- 已裁定：F 条

---

## 🔴 阻塞条目（A 条，立即处理）

### Q[N] | [角色] | [模块] | [日期]
**问题**：{原文}
**回答候选**：{基于 PRD 起草的草稿}
**用户决策**：☐ 采纳草稿 ☐ 修改 ☐ 升级讨论

---

## 待裁定条目（按类型聚合）

### 类型：业务判断（B 条）

#### Q[N] | [角色] | [模块] | [日期]
**问题**：{原文}
**已查上下文**：
- 主 PRD `{章节}`：{相关原文}
- 主 PRD 第 1 章「考虑过但放弃的设计」段：{相关上下文}
- 硬约束 {N}：{是否冲突}

**回答候选 A**：{方案 A + 理由 + 影响}
**回答候选 B**：{方案 B + 理由 + 影响}（如有 2 个备选）
**用户决策**：☐ 选 A ☐ 选 B ☐ 自填 ☐ 升级讨论

#### Q[N+1] ...

### 类型：跨角色冲突（C 条）

...（同上格式）

### 类型：歧义（D 条）

...

### 类型：缺漏（E 条）

...

---

## 已裁定条目摘要（F 条，不展开）

| Q | 角色 | 模块 | 裁定日期 | 结论摘要 |
|---|---|---|---|---|
| ... | ... | ... | ... | ... |

---

## FAQ 候选（重复 ≥ 3 次）

### 候选 1：{重复主题}
- 涉及问题：Q3 / Q7 / Q12（共 3 次）
- 建议固化到主 PRD 第 1 章 FAQ 段：`{建议的 FAQ 标题 + 标准答案}`

---

## 建议的下一步

1. 处理 🔴 阻塞条目（{A} 条）→ 估时 ~{N} 分钟
2. 批量裁定业务判断条目（{B} 条）→ 估时 ~{N} 分钟
3. ...
4. 把 FAQ 候选搬到决策上下文（{个数}）

总耗时预估：{总分钟数}（vs. 直接看原始文件大约 {2-3 倍} 分钟）
```

### 步骤 7：用户确认后写入 OPEN_QUESTIONS.md

用户对每条 Q 的"回答候选"做出选择（采纳 / 修改 / 否决）后，本 skill 自动批量回写到 `OPEN_QUESTIONS.md`：

- 在每条 Q 的"当前状态"行改为：`用户已裁定（YYYY-MM-DD）：{结论}`
- 移除 🔴 阻塞 标记（如已解决）

如用户对某条 Q 选"自填"或"升级讨论"，本 skill 不自动写入，提示用户手动处理或开新对话讨论。

## 三、强制规则

1. **回答候选是草稿，不是终答**：必须等用户确认才写入 OPEN_QUESTIONS.md，禁止自动覆盖
2. **必须读 PRD / 决策上下文再起草**：禁止凭直觉编答案
3. **硬约束冲突必须显式标注**：不能默认建议绕开硬约束
4. **保留所有原始 Q 内容**：回答时只追加裁定，不修改原始问题描述
5. **多个 OPEN_QUESTIONS.md 并行扫时，按"项目 / 需求"分组输出**，不混在一起
6. **已裁定条目仅汇总数量**：不展开重复（除非用户特别要求"看历史答疑"）
7. **跨角色冲突类问题**：除给回答候选外，必须建议产品总监同步通知所有相关角色

## 四、与其他工具的协作

| 工具 | 关系 |
|---|---|
| `prd-writer-agent` | 起草回答候选时读 PRD（间接依赖） |
| `demand-decision-agent` | 起草回答候选时读决策上下文（间接依赖） |
| FAQ 升级 → 主 PRD 第 1 章 FAQ 段 | 升级 ≥ 3 次的重复问题 |
| 硬约束冲突 → 决策表层 | 如某条问题揭示硬约束需要调整，建议升级到决策表 |

## 五、未来扩展（待实施）

- **多项目并行扫描**：当前需求规模小，单项目即可；未来需求 ≥ 5 个时支持跨项目聚合
- **答疑模板库**：常见问题的标准答案库，让 AI 起草更快更一致
- **协作健康度指标**：本周答疑时长 / 平均待答时间 / 阻塞条目消化速度等

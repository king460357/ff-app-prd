# open-questions-triage 使用示例

## 示例 1：日常批量答疑（用户主流场景）

### 用户输入

> "看下 OPEN_QUESTIONS"

或更具体：

> "扫一下 {批次}/{需求名称} 的 OPEN_QUESTIONS"

### Skill 执行步骤

1. 定位文件：`docs/{YYYYMMDD}/PRD/{需求名称}/OPEN_QUESTIONS.md`
2. 解析每条 Q 的字段
3. 排序 + 分类
4. 读主 PRD / confirmed-decisions
5. 起草回答候选
6. 输出报告：

```markdown
# OPEN_QUESTIONS 分诊报告

**扫描范围**：docs/{YYYYMMDD}/PRD/{需求名称}/OPEN_QUESTIONS.md
**扫描时间**：YYYY-MM-DD HH:MM
**总问题数**：6
- 🔴 阻塞：1 条
- 业务判断：2 条
- 跨角色冲突：1 条
- 歧义：1 条
- 缺漏：1 条
- 已裁定：0 条

---

## 🔴 阻塞条目（1 条，立即处理）

### Q3 🔴 阻塞 | 后端 | {模块编号} {模块名} | YYYY-MM-DD
**问题**：第三方服务返回的 `error_code = no_authorization_v2`（v2 版后缀）在我们的映射表里不存在。是要按 `no_authorization`（v1）处理还是新建一条？等运营回复阻塞了我接口实现。

**已查上下文**：
- 主 PRD {章节编号}：原因映射表后台允许运营随时新增 error_code
- 主 PRD 第 1 章「考虑过但放弃的设计」段：未提及 v2 版本

**回答候选**：
按 v1 `no_authorization` 处理（继承中文原因 + 修改建议）。同时通知运营在后台**先新建一条 `no_authorization_v2` 映射**（与 v1 同结论），避免后续再次出现。

后端可继续实现接口，按以下逻辑：

```python
if error_code in mapping_table:
    return mapping_table[error_code]
elif error_code.startswith('no_authorization'):  # 兼容 v1 / v2 / vN
    return mapping_table['no_authorization']
else:
    return mapping_table['default']
```

**用户决策**：☐ 采纳草稿 ☐ 修改 ☐ 升级讨论

## 待裁定条目（按类型聚合）

### 类型：业务判断（2 条）

#### Q1 | 数据库 | {模块编号} {模块名} | YYYY-MM-DD
**问题**：自动沉淀阈值"24h ≥ 3 次拦截 + ≥ 2 用户"中的"24h"是滚动 24 小时还是自然日？
**我倾向方案**：滚动 24 小时（更敏感）

**已查上下文**：
- 主 PRD {章节编号}：未明确
- 主 PRD 第 1 章 FAQ 段 Q1：阈值可调，但"24h"含义未提
- 硬约束 1：运营负担 ≤ 40 分钟/日 → 滚动 24h 触发更频繁，运营审核压力可能上升

**回答候选 A**：滚动 24 小时（数据库 AI 倾向）
- 优点：发现更敏感
- 缺点：审核压力可能超标，违反硬约束 1
- 实现：用 Redis sorted set 维护 24h 滑动窗口

**回答候选 B**：自然日（00:00-23:59）
- 优点：审核压力可控（每日有上限）
- 缺点：跨日的相邻时段问题发现延迟
- 实现：日终 Cron 计算前 24 小时

**用户决策**：☐ 选 A ☐ 选 B ☐ 自填 ☐ 升级讨论

### 类型：跨角色冲突（1 条）

#### Q4 | 前端 ↔ 后端 | {模块编号} {模块名} | YYYY-MM-DD
**问题**：后端的预检接口返回 `{ hits: [...], suggestions: [...] }` 两个数组分开。前端意图书希望合并为 `{ items: [{hit, suggestion}] }` 一一对应的列表，因为 UI 上每个命中卡片紧挨着自己的建议。这个改要后端配合，已经写到 OPEN_QUESTIONS。

**已查上下文**：
- 主 PRD {交互章节}：每个命中卡片紧挨着建议（UI 视角）
- 主 PRD 第 1 章前端 / 后端易犯偏差段：前端不应自行设计接口，后端不应忽略前端 UI 需求

**回答候选**：后端调整为合并格式 `{ items: [{hit, suggestion}] }`，理由：
1. 与 UI 设计的"卡片配对显示"一致（主 PRD {交互章节}）
2. 避免前端组装时出错（如长度不匹配）
3. 后端工作量增加 < 30 分钟

**通知通知**：✋ 此决策影响后端接口 + 前端组件，建议通知后端 AI 同步调整主 PRD 第 8.3 节接口契约。

**用户决策**：☐ 采纳草稿 ☐ 修改 ☐ 升级讨论

### 类型：歧义（1 条）

#### Q2 | 前端 | {模块编号} {模块名} | YYYY-MM-DD
**问题**：拦截弹窗的"按建议修改"按钮，PRD 说"关闭弹窗回到任务编辑页"，但没说原任务是否保留。是保留任务等用户编辑完再提交，还是丢弃任务让用户重新提交？

**已查上下文**：
- 主 PRD {按钮交互章节}：未明确
- 主 PRD {状态清单章节}：任务有"已取消"状态，但触发条件未列拦截

**回答候选**：保留任务（不取消）。理由：
1. 用户如果是想"按建议修改"，丢弃任务会导致已上传的资产 / 已写的提示词都丢失，体验差
2. 主 PRD 主流程的"拦截 → 修改 → 重试"闭环也是基于"同一任务"
3. 任务保留为 `pending_user_edit` 状态（新增），用户提交后改为正常 `processing`

**用户决策**：☐ 采纳草稿 ☐ 修改 ☐ 升级讨论

### 类型：缺漏（1 条）

**问题**：极简申诉走客服群剪贴板，但 PRD 没说"剪贴板复制后是否在数据库留痕"。如果不留痕，未来想统计"客服消化哪些拦截案例"会缺数据。

**已查上下文**：

## 示例 2：跨批次并行扫描

### 用户输入

> "扫一下所有批次的 OPEN_QUESTIONS"

### Skill 执行

1. Glob 扫描 `docs/*/PRD/*/OPEN_QUESTIONS.md`
2. 找到例如：
   - `docs/{批次A}/PRD/{需求1}/OPEN_QUESTIONS.md`（6 条问题）
   - `docs/{批次A}/PRD/{需求2}/OPEN_QUESTIONS.md`（3 条）
   - `docs/{批次B}/PRD/{需求1}/OPEN_QUESTIONS.md`（1 条）
3. 按"批次 / 需求"分组输出，每组一份子分诊报告
4. 顶部加跨批次摘要：

```markdown
# 跨批次 OPEN_QUESTIONS 分诊总览

**扫描范围**：3 个需求目录
**总问题数**：10（🔴 阻塞 1 / 待裁定 9）

| 批次 / 需求 | 总数 | 🔴 阻塞 | 待裁定 |
|---|---|---|---|
| {批次A} / {需求1} | 6 | 1 | 5 |
| {批次A} / {需求2} | 3 | 0 | 3 |
| {批次B} / {需求1} | 1 | 0 | 1 |

---

（以下按批次/需求展开各自分诊报告，格式同示例 1）
```

## 示例 3：FAQ 升级触发

### 当 OPEN_QUESTIONS.md 中重复出现 ≥ 3 次的问题

```markdown
## FAQ 候选（重复 ≥ 3 次）

### 候选 1：第三方服务 error_code 不在映射表怎么办
- 涉及问题：Q3（已裁定）/ Q8（已裁定）/ Q14（待裁定）
- 三次问题都是新 error_code 出现
- 建议固化到主 PRD 第 1 章 FAQ 段：
  > **Q：第三方服务返回的 error_code 不在映射表怎么办？**
  > A：按前缀匹配兜底（如 `no_authorization*` 都映射到 v1 处理）；同时通知运营在后台新建该 error_code 的映射。详见 SKILL.md 对应章节示例代码。
```

用户确认后，本 skill 把这条搬到主 PRD 第 1 章 FAQ 段；下次类似问题再来时，开发端 AI 读主 PRD 就能自答，不再写入 OPEN_QUESTIONS。

## 关键提示

1. **回答候选是省时间用的**：不替代用户判断；用户对所有候选有最终决定权
2. **主 PRD 已覆盖的问题**：本 skill 会指出"开发端漏看了主 PRD `{章节}`"，节省用户解释成本
3. **硬约束冲突必显式标注**：避免用户因为忙而漏过冲突
4. **跨角色冲突会建议同步通知**：避免改了一边没改另一边

# PRD Writer — 产品需求文档生成 Skill

## 核心定位

帮助有产品想法但不熟悉 PRD 写法的用户，把想法系统化地转变为**可执行**的需求文档。

**核心交付承诺**：产出的 PRD 必须让 5 类角色都能直接用：

- **产品经理**：方向清晰、决策依据完整
- **研发工程师**：知道做什么、改什么、对接什么
- **UI/UX 设计师**：知道页面结构、状态变化、交互反馈
- **QA 测试**：能写出测试用例
- **运营**：能理解功能价值和用户路径

---

## 核心设计思路

- **先验证方向，再展开细节**——方向错了，细节都是浪费
- **分两阶段交付**：第一版对齐方向，第二版落地细节
- **三视角切换**：用户需求 → 商业可行 → 技术落地
- **主动补盲区**：交互细节、状态机、数据规范、文案风格——小白用户不会想到的，主动补上
- **唯一规则源**：同一规则只在一个章节定义，其他章节引用，不重复啰嗦
- **多角色可读**：产品/研发/UI/QA/运营，五个角色都要能看懂

---

## 第零步：判断用户是否适合用这个工具

### ① 确认模式偏好（优先询问）

> "你是想要完整引导式（我会分步骤帮你对齐方向再落地细节，质量更高，需要几轮对话），还是快速草稿模式（我先出一版草稿，你再告诉我哪里需要调整）？"

- 用户选**完整模式** → 按正常流程走（三视角诊断 → 概念版 → 落地版）
- 用户选**快速草稿** → 收集最基本信息后直接出草稿，跳过多轮诊断，后续按需细化
- 用户没有明确偏好 → 默认走完整模式

### ② 了解核心想法

> "在开始之前，能先跟我说说你这个产品/功能的核心想法是什么吗？哪怕一两句话就行——你想解决什么问题，大概想用什么方式解决？"

**判断标准：**

| 用户的回答 | 判断 | 处理方式 |
|-----------|------|---------|
| 能说出「谁的什么问题，用什么方式解决」 | ✅ 有核心逻辑 | 进入第一步 |
| 描述模糊但有方向，如「我想做个帮人记账的 App」 | ⚠️ 有苗头 | 追问几个问题帮他明确核心逻辑，再进入第一步 |
| 完全没有方向，如「我想做个 App，你帮我想做什么」 | ❌ 暂不适合 | 不要直接拒绝——先问：「你生活或工作中，有没有哪件事让你觉得特别麻烦，一直没找到好的工具？」帮用户从真实痛点出发找到方向，再进入第一步 |

---

## 识别用户当前的阶段

| 用户的情况 | 进入模式 |
|-----------|---------|
| 有想法，还没有文档 | → **模式 A**：从零引导，分两版生成 |
| 已有需求文档，想检查/改进 | → **模式 B**：评估 + 改进 |
| 已有文档，想追加新需求或补充细节 | → **模式 C**：增量融合更新 |
| 情况模糊（既有想法，又有零散草稿） | → **优先进模式 B**，评估现有内容，再决定是否用模式 A 补充未覆盖的部分 |

---

## 模式 A：从零引导，分两版生成

### 第一步：三视角诊断（方向对齐前的必做项）

每次只问 1-2 个问题，自然推进，直到三个视角都有清晰答案。

#### 视角 1：用户角度——这个需求真实存在吗？
- 你的目标用户现在是怎么解决这个问题的？（他们用什么工具/方式）
- 他们现有的方式有什么不够好？你的产品比现有方式好在哪里？
- 你有没有跟真实用户聊过这个痛点？他们是怎么描述的？

诊断目标：确认需求真实存在，而不是「我觉得用户应该需要」的假设。

#### 视角 2：商业角度——这件事值得做吗？
- 这个产品怎么赚钱，或者对你有什么商业价值？（付费订阅 / 广告 / 内部降本）
- 市场上有没有类似的产品？你和他们最大的差异是什么？
- 你的目标用户规模大概是多少？

诊断目标：确认有持续运营的商业逻辑，而不是白忙活。

#### 视角 3：开发角度——这件事做得出来吗？
- 这个产品最核心的技术能力你现在有吗，还是依赖第三方？
- 有没有什么你觉得实现起来最难的部分？
- 如果先做一个最小可用版本（MVP），哪些功能是必须有的？

诊断目标：确认方向上没有技术死角，能找到一个可以先跑起来的最小版本。

#### 产品形态选择——用什么载体来承载这个产品？

**如果用户已经说出了产品形态**，先温和确认他是否真的想清楚了：「你说的 App，是需要下载安装的原生 App，还是手机浏览器打开的网页，或者微信小程序？大概为什么选这个形式？」

**如果用户没有明确产品形态**，根据三视角信息主动推荐，不要抛出选项让用户自己判断。

| 形态 | 适合场景 | 核心优势 | 主要限制 |
|------|---------|---------|---------|
| **网页 Web App** | 功能复杂、需要大屏操作、B 端工具 | 开发最快，跨平台，SEO 友好 | 移动端体验弱于原生，推送受限 |
| **微信小程序** | 目标用户主要在中国，强依赖社交传播 | 微信生态红利，无需下载 | 受微信平台约束，出海不适用 |
| **原生 App** | 强依赖设备能力（相机/GPS），高频使用，有明确变现路径 | 体验最好，推送稳定，可离线 | 开发成本高，审核周期长 |
| **AI Skill / 插件** | 功能本质是增强某个已有工具，而不是独立产品 | 开发成本极低，利用宿主平台流量 | 强依赖宿主平台，自由度受限 |

推荐时说明理由框架：① 目标人群用什么设备/渠道发现产品；② 核心功能有没有设备能力需求或离线需求；③ 团队开发成本和上线速度约束；④ MVP 优先——优先选「最快验证需求」的形态。

诊断完成后，先口头总结你理解的产品方向，让用户确认，再进入下一步。

---

### 第二步：输出【第一版 · 产品概念文档】

**第一版唯一目的：对齐方向。** 保持简洁，不展开细节。

```markdown
# 【产品名称】（暂定）

## 一句话定位
> 这是一个给【目标用户】用的【产品形态】，帮他们【解决什么问题】。
> 与现有方案相比，核心差异是【差异化优势】。

## 产品形态
- **当前选型**：【网页 Web App / 微信小程序 / 原生 App / AI Skill】
- **选择理由**：（简要说明为什么选这个形态）
- **阶段策略**（如有）：（如「先做网页版验证需求，后续考虑出 App」）

## 目标用户
- 核心用户画像（1-2 类，说清楚是谁、有什么特征）
- 他们的核心痛点
- 为什么会选择这个产品，而不是继续用现有方式

## 产品价值
- 用户获得的价值
- 商业价值/变现逻辑

## 核心功能方向（只列方向，不展开细节）
- 功能方向 1
- 功能方向 2
- 功能方向 3

## 不做什么（边界）
- 明确列出哪些需求超出本产品范围，以及为什么不做

## 已知风险
- 风险 1（方向/技术/市场层面）
- 风险 2
（如诊断阶段未发现明显风险，写「暂未发现方向性风险，进入落地阶段后持续关注」）

## 待确认问题
- 还需要用户回答才能继续推进的问题
```

输出后询问：「这份概念版符合你的预期吗？产品定位、目标人群、核心方向——有没有哪里感觉不对？没问题了我们再展开细节。」

- 用户满意 → 进入第三步
- 用户觉得不对 → 根据反馈修改，重复此步骤直到双方对齐（不因用户说「差不多」就跳过）

---

### 第三步：输出【第二版 · 落地需求文档】

⚠️ **只有概念版得到用户明确确认后，才能进入这一步。**

落地版目标：让文档能被 AI 或开发者直接拿去实现。信息不足时标 `[待补充]`，**不能跳过或用模糊语言带过**。

⚠️ **落地版在聊天窗口输出完毕后，必须执行文件落盘（见"📤 文件落盘"章节），将 PRD 保存到项目内 `docs/prd/{项目名}/{需求名称}/` 目录。禁止写入用户桌面 `~/Desktop/`。**

#### 文档结构（PRD 必含 7 大类，全部强制）

```
1. 产品概述              ← 包含【背景 & 目标】（必含 1）
2. 目标用户与使用场景    ← 【用户场景】（必含 2）
3. 核心用户动线          ← Mermaid 流程图，主流程 + 1-2 个异常分支
4. 功能清单              ← 【功能描述】总览（必含 3 之一）；树状结构 + 优先级 🔴核心/🟡重要/⚪未来规划
   4.1 关键页面线框图    ← 【效果图】之 ASCII（必含 5 之一）
   4.2 UI 设计图提示词   ← 【效果图】之 HTML 实物 UI 渲染英文提示词（必含 5 之一）
5. 功能详细描述          ← 【功能描述】+【交互说明】+【验收标准】每个 🔴 核心功能独立写一节，12 项必填
6. 文案规范              ← 双受众分开定义
7. 非功能性需求
8. 影响范围矩阵          ← 【影响范围】给研发的总览章节（必含 7），按前端/后端/数据库/第三方/上线顺序聚合
9. 技术选型建议          ← 面向无技术背景用户；已有选型则跳过
10. 待确认问题
```

**七大必含项与章节映射**（用户层面命名 → PRD 章节位置）：

| # | 必含项 | 落在 |
|---|---|---|
| 1 | 背景 & 目标 | 第 1 章产品概述（含背景、问题、目标指标） |
| 2 | 用户场景 | 第 2 章目标用户与使用场景 |
| 3 | 功能描述 | 第 4 章功能清单（总览）+ 第 5 章功能详细描述（细节） |
| 4 | 交互说明 | 第 5 章 12 项必填中的「按钮交互/弹窗规则/状态清单/异常处理/联动规则」 |
| 5 | 效果图（必须） | 4.1 ASCII 线框图 + 4.2 UI 设计图英文提示词；两者都要给；**凡须原型展示的页面**还须满足下文 **「原型图强制展示（页面级）」**（ASCII / Figma 锚点 / 导出图 / HTML 实物 UI 至少其一，禁止纯文字替代） |
| 6 | 验收标准（必须，可测试） | 第 5 章 12 项必填的第 12 项 |
| 7 | 影响范围（前端/后端/数据库） | **第 8 章影响范围矩阵**（独立章节，研发第一眼能看全） |

#### 5. 功能详细描述——12 项强制清单（每个 🔴 功能必须全部完成）

📖 每一项的具体要求和填写模板：参见 `interaction_detail_standard.md`

| # | 项目 | 强制要求 |
|---|------|---------|
| 1 | 功能说明 | 解决什么问题、面向谁、用于什么场景 |
| 2 | 触发条件 | 什么时候启动这个功能 |
| 3 | 入口位置 | 用户从哪里能进入 |
| 4 | 页面结构 | 整体布局描述 |
| 5 | UI 示意 | ASCII 线框图（参见 `ui_ascii_template.md`）；**须原型展示的页面**按 **「原型图强制展示（页面级）」** 补齐 Figma 锚点或导出图或 HTML 实物（与项目双产物策略一致） |
| 6 | 按钮交互 | 每个按钮 6 项必填（见下） |
| 7 | 弹窗规则 | 每个弹窗 5 项必填（见下） |
| 8 | 状态清单 | 空状态/加载中/成功/失败/部分失败/置灰 6 类全覆盖 |
| 9 | 异常处理 | 空内容/超长/网络异常/无权限/并发/格式不符 6 类边界 |
| 10 | 联动规则 | 与其他模块的同步关系（数据来源/同步时机/冲突处理） |
| 11 | 数据规范 | 每个字段：字段名（中英文）/类型/长度/必填/默认值/校验规则 |
| 12 | 验收标准 | 可观察、可判定，不允许"流畅、美观"等模糊词 |

**按钮交互 6 项**：① 默认是否可点击 ② 点击后是否弹窗 ③ 点击后页面状态如何变化 ④ 是否可重复点击 ⑤ 成功后去哪里 ⑥ 失败后提示什么

**弹窗规则 5 项**：① 标题 ② 正文 ③ 按钮文案 ④ 点击取消的结果 ⑤ 点击确认的结果。危险操作（删除/清空/扣款等）必须二次确认 + 确认按钮用警告色。

#### 4.2 UI 设计图提示词（用于 HTML 实物 UI 渲染）

**第一步：定义视觉设计系统（全局生效，所有页面共用）**

在生成各页面提示词之前，先收集或推断以下信息，信息不足时给出推荐默认值并标注 `[AI 推荐]`：

- 目标平台：移动端 iOS / 移动端 Android / 桌面 Web / 响应式
- 设计风格：现代扁平 / 轻拟物 / 毛玻璃 / 极简主义 / 商务专业 / 活泼消费
- 配色方案：主色（Primary HEX）/ 背景色 / 卡片背景 / 主文字色 / 次文字色
- 模式：浅色（Light）/ 深色（Dark）/ 两者都要
- 字体风格 / 圆角风格 / 阴影风格

**第二步：为每个关键页面输出一段英文图像提示词**

提示词格式：
```
UI design rendering of [产品名] [页面名称], [平台类型] app screenshot.
Design style: [风格英文描述]
Color scheme: Primary [HEX], Background [HEX], Cards [HEX], Text [HEX]
Mode: [Light / Dark]
Layout structure: [用英文描述页面布局，参照 4.1 ASCII 线框图转为自然语言]
Key UI elements: [3-5 个最重要的 UI 元素具体描述]
Content to display: [页面上显示什么样的示例内容]
Typography: [字体风格]
Quality: high-fidelity UI rendering, pixel-perfect, professional app design, no watermark
```

- 有几个核心页面就输出几段提示词（通常 3-5 个）
- 所有页面共用同一套视觉设计系统，只替换布局结构和关键元素
- 4.1 ASCII 线框图（给开发者看结构）和 4.2 图像提示词（给图像模型看视觉）互补，**两者都要输出**
- 配色未指定时按产品类型推荐（电商→暖橙/红；工具→蓝/灰；健康→绿/白；金融→深蓝/金）并标 `[AI 推荐]`

#### 原型图强制展示（页面级，2026-05-13 起）

> **硬规则**：凡 PRD 中涉及「需评审布局 / 信息架构 / 多状态界面」的**页面或等价界面**（含独立 Tab 页、全屏/半屏模态、Bottom Sheet、关键空态/错误态、多版本 Before/After），均视为**需要用原型图展示的页面**——**必须**用原型图级材料展示出来，**禁止**仅用纯文字段落描述顶替。

**「原型图」可接受形态**（至少满足其一；**markdown 真理源**中须可直接查看或可打开链接）：

| 形态 | 适用 | 要求 |
|------|------|------|
| **A. ASCII 线框** | 默认、无设计稿时 | 写入第 **4** 章与对应 **5.x** 节**内联**紧贴文字；有改版须 **Before / After**；禁止把所有线框堆在文档末尾而无章节锚点 |
| **B. Figma / 设计工具锚点** | 已有设计文件时 | PRD 内写明 **文件 URL + Page 名 + Frame 名**（或 `node-id=`），评审可直达对帧 |
| **C. 导出图** | 高保真为事实源时 | 落在本需求目录 `assets/` 或 `_prototypes/` 下，Markdown 用 `![](相对路径)` **嵌入该页面描述节** |
| **D. HTML 派生 PRD** | 项目启用 `.md` + `.html` 双产物时 | `.html` 中该页须 **HTML/CSS 实物 UI**（见 `html_ui_rendering_standard.md`），事实与 `.md` 一致 |

**与第 5 章关系**：每个 🔴 模块若涉及独立屏/弹层，在 **5.x.5 UI 示意**中必须出现对应原型（通常即 ASCII；若该屏已在第 4 章画过，可写「见 4.x.x」但**不得省略图**）。

**自检（产出前必跑）**：枚举本文档所有「页面 / 弹层」标题 → 逐项核对是否具备 A/B/C/D 之一；任一项缺失 → **PRD 不完整，须返工**。

#### 研发免追问 PM（落地版硬目标，2026-05-13 起）

> **定义**：落地版 PRD 交付研发时，应达到「**研发/QA 无需再找产品经理澄清业务逻辑即可开工与写用例**」；技术实现细节（HTTP 路径、DDL、类设计）仍由研发在技术 spec 中深化。

**必须满足（缺一返工）**：

1. **身份 / 状态真值**：有 **状态枚举 + 状态转移表**（事件 → 下一状态 → 客户端/服务端动作），或与 **能力矩阵表**（角色 × 功能 × 允许/禁止）等价且可交叉校验。
2. **鉴权与写路径**：明确每种身份下调哪些接口、**禁止**哪些客户端行为（防绕过）；写接口无 token 时的用户可见结果。
3. **接口业务契约**：至少一张 **「接口意图」表**（无 URL 亦可）：鉴权、请求/响应**业务字段**、幂等要求；错误场景有**用户文案**（PM 草稿即可）。
4. **第三方/SDK**：有 **开关矩阵**（按身份或按阶段 ON/OFF），访客/合规敏感路径不得「依赖研发临场判断」。
5. **开放问题收口**：不得把产品歧义长期留在「待确认」；须 **拍板默认策略** 或 **远程开关默认值** 或 **单列法务声明且不阻塞实现**（实现按默认，法务冲突再走变更单）。
6. **原型图**：仍须满足上文「原型图强制展示」自检。

**与 PM 边界关系**：不要求 PM 写 DDL/具体错误码；**要求** PM 把业务分叉写全，避免研发猜。

#### 8. 影响范围矩阵（必含项 7，给研发的总览章节）

研发拿到 PRD 后，第一眼应能看清「我要改哪些前端、哪些后端、哪些表」。详细字段定义见各模块 5.x.9，详细接口逻辑见 5.x.8；本章只做聚合总览。

强制包含 7 个子节：

| 子节 | 内容要求 |
|---|---|
| 8.1 模块 × 端 总览 | 一张表，每行一个 🔴/🟡 模块，列：优先级 / 前端 / 后端 / 数据库 / 第三方依赖 / 工作量预估（小/中/大）|
| 8.2 前端影响详情 | 列出每个前端页面/组件是「新增」还是「修改」，定位到端（用户/运营/客服） |
| 8.3 后端影响详情 | 列出每个服务的「新增接口/任务」与「修改的现有逻辑」 |
| 8.4 数据库影响详情 | 每张表标注「新增/修改/仅读」，含字段范围、引擎、预估量级 |
| 8.5 第三方依赖 | 每个外部依赖的用途 + 风险/注意事项 |
| 8.6 模块间数据流依赖 | Mermaid 图：模块 ↔ 表 / 模块 ↔ 第三方的依赖箭头 |
| 8.7 上线依赖顺序 | 列出 N 个步骤的前后置关系，含必须前置项 |

写法要点：
- 不重复 5.x.9 / 5.x.8 的细节，仅聚合总览
- 表格优先于段落（飞书直接复制可读）
- 「修改的现有逻辑」要写清是哪段代码/哪个接口的什么行为变化

## 多角色 AI 服务（主 PRD 自包含原则）

PRD 不只给"开发"看——它服务的是 **5 类专业 AI 角色**：前端、后端、数据库、QA、UI。每类关注点不同。**默认主 PRD 自身已自包含所有角色所需信息**——通过 frontmatter `ai_context` 角色路由 / 第 1 章决策上下文 / 第 4.3 节 UI 提示词 / 第 5 章每模块「目标读者」标签等结构化章节直接服务 5 类角色，不再单独建配套文件。

⚠️ **PM 边界硬约束**（2026-05-11 起 — 完整规则见 `prd_vs_techspec_boundary.md`）：
- PRD 是 **业务真理源**，不是 **技术实现手册**
- 8.3.1 PM 只写「错误场景 + 用户提示文案」，**具体错误码命名** 由研发深化
- 8.4 PM 只写「业务字段需求」，**表设计 / 字段类型 / 索引** 由研发深化
- 8.4.1 完整 DDL **由研发在技术 spec 中编写**，PRD 不含 CREATE TABLE 语句
- 判断越界 3 问：① 换技术栈成立吗 ② 研发能直接复制吗 ③ 是用户视角还是系统视角？

### 标准交付物清单

每份新 PRD 产出时，必须产出 3 份资产（双产物 + 目录引导 README）：

| # | 资产 | 文件名 | 性质 |
|---|---|---|---|
| 1 | 主 PRD（含 frontmatter ai_context + 7 项必含 + 第 1 章决策上下文 + 第 4.3 节 UI 提示词 + 第 5 章每模块目标读者标签 + 第 8 章影响范围）— ⚠️ **PM 边界约束**：8.3.1 只写「错误场景 + 用户文案」（不写具体错误码命名）/ 8.4 只写「业务字段需求」（不写表设计）/ 8.4.1 由研发在技术 spec 中编写（PM 不写完整 DDL）| `{YYYYMMDD}_{需求名}_PRD.md` | **必有**（真理源）|

## 模式 B：评估 + 改进已有文档

收到用户的需求文档后，按以下四层 checklist 逐项检查，每项给出 ✅ / ❌ / ⚠️：

**方向层**
- [ ] 有明确的目标用户和核心痛点
- [ ] 有清晰的一句话产品定位
- [ ] 有明确的产品形态（Web / 小程序 / App / Skill）及选型理由
- [ ] 有商业价值或变现逻辑
- [ ] 有明确的「不做什么」边界

**结构层**
- [ ] 有核心用户动线（最好有流程图，至少有文字描述）
- [ ] 有结构化功能清单（带优先级）
- [ ] 有关键页面布局线框图
- [ ] 每个核心功能有独立详细描述（没有合并省略）

**细节层**
- [ ] 每个交互元素列出了全部状态（含加载中/失败/禁用/空状态）
- [ ] 边界条件已覆盖（空内容、超长、网络异常、无权限、并发）
- [ ] 有交互细节（操作反馈、危险操作确认、空状态引导、失败引导）
- [ ] 有数据规范（字段名、类型、长度、必填、默认值）
- [ ] 有面向终端用户的文案规范，且与开发文档分开

**质量层**
- [ ] 不存在「原文保留/此处略/参考上文/后续补充」等占位语
- [ ] 没有同一规则在多处重复定义（违反唯一规则源）
- [ ] 英文术语都有中文说明
- [ ] 代码块只用于 Mermaid/ASCII/字段名/接口路径，没有滥用
- [ ] 状态命名使用中文（未开始/生成中/已完成/生成失败/部分失败 等）

## 模式 C：增量融合更新

**核心原则**：融合而不是覆盖，更新而不是重写。

1. 让用户提供现有文档
2. 明确用户想追加/修改的内容
3. 定位影响范围（通常需同步更新功能清单、用户动线、功能详细描述）
4. 执行融合，用 `【本次更新】` 标注所有改动
5. **融合后必检 3 项**：
   - 用户动线是否同步更新（流程图是否加了新分支？）
   - 新功能是否补充了交互细节、状态、异常、验收
   - 是否触发了**唯一规则源冲突**（新需求里写的规则，原文档其他章节是否已定义过？如有冲突需指出）

## 🚨 通用红线（七大铁律，必须遵守）

1. **分阶段，不跳跃**：概念版没得到确认前，不展开落地细节
2. **三视角必须过**：用户/商业/开发三个角度都必须有基本清晰的答案
3. **主动补盲区**：交互细节、数据规范、文案规范——主动补，不等用户问
4. **多受众意识**：AI/开发者（精确可执行）vs 终端用户（友好简洁），两者语言风格不同，分开定义
5. **融合不覆盖**：追加需求时不丢失原有内容，用 `【本次更新】` 标注改动
6. **宁可标 `[待补充]` 也不编造**：信息不足时明确标出，不靠猜测填内容
7. **唯一规则源**：同一规则只在一个章节完整定义，其他位置只引用，绝不重复完整描述

## 🚫 严格禁止清单（违反 = 失败输出）

### 禁止占位语

当用户要求"完整需求 / 可以直接复制 / 不要缩略"时，**禁止**使用以下表达：

| 禁止 | 替代 |
|------|------|
| 原文保留 | 完整重写一遍 |
| 此处略 | 完整写出 |
| 参考上文 / 同上 | 重新展开（或加引用："详见第 X 章 X.Y 节"） |
| 后续补充 | 标 `[待补充]` 并说明缺什么 |
| 原表保留 / 你原来的内容不变 | 完整复制原表过来 |
| 暂时无法 / 可能 / 应该 / 大概 | 给出明确判断或标 `[待确认]` |

### 禁止滥用代码块

📖 详细规则：`output_format_standards.md`

**只允许**用代码块的内容：Mermaid 流程图、ASCII 界面示意图、路由/接口/字段名、命令/脚本/JSON/SQL

**禁止**用代码块的内容：功能说明、交互说明、按钮行为、状态解释、异常处理、验收标准、修改原因、产品注解

### 禁止英文术语无中文说明

📖 详细规则：`chinese_term_standard.md`

- 正文优先使用中文；英文术语**首次出现**必须加中文说明，如：`prompt（提示词）`、`fallback（兜底）`、`asset（资产）`
- 状态名优先使用中文：未开始 / 生成中 / 已完成 / 生成失败 / 部分失败 / 已保存 / 未保存 / 只读 / 已锁定 / 可点击 / 置灰
- 字段表（接口字段）可保留英文枚举，但必须有中文说明列

### 禁止重复定义同一规则

📖 详细规则：`deduplication_rules.md`

- 规则章节写完整规则；场景章节只写差异；UI 章节只写界面表现；流程图只做视觉表达；验收只写是否达标

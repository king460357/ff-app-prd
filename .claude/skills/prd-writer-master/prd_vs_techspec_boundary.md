# PRD vs 需求 Spec vs 技术 Spec 边界（2026-05-11 起 — PM 不越界硬约束）

> 本文件给 prd-writer-agent 用——产出 PRD 时**严格按本文件边界**确定写到哪一层、停在哪里。
>
> **核心原则**：PRD 是**业务真理源**，不是**技术实现手册**。PM 越界写代码细节 = 浪费时间 + 跟研发抢工作 + 增加 PRD 维护成本。

---

## 1. 三层文档定义

| 层 | 文档名 | 核心问题 | 受众 | 责任人 | 颗粒度 |
|---|---|---|---|---|---|
| L1 | **PRD** | **为什么做 + 做什么** | 全员（含老板 / 跨团队） | **PM** | 界面 / 行为 |
| L2 | **需求 Spec** | **业务怎么运转** | 实施团队（研发 + QA） | PM 或研发 | 业务规则 / 字段 / 状态机 |
| L3 | **技术 Spec** | **代码怎么实现** | 研发团队 | **研发架构师 / Tech Lead** | 模块 / 接口 / DDL / 部署 |

**字节系 / 主流互联网公司实践**：PRD 内化 L1 + L2（PM 写），技术 Spec 单独由研发写。

---

## 2. PM 写 vs 研发写 — 边界硬约束

| 章节 | 内容 | PM 写到什么程度 | 研发深化什么 |
|---|---|---|---|
| **1. 产品概述** | 背景 / 问题 / 目标 / 上线指标 | ✅ **PM 全写** — 业务视角说清"为什么做这件事" | 不参与 |
| **2. 目标用户与场景** | ≥ 3 个具体场景 | ✅ **PM 全写** — 用户视角说清"怎么用这个功能" | 不参与 |
| **3. 核心用户动线** | Mermaid 流程图 | ✅ **PM 全写** — 业务流程图 | 不参与 |
| **4. 全局视觉** | 视觉规范 + 整体一图速览 | ✅ **PM 全写** — UI 视觉约束 | 不参与 |
| **5.x.1-5.x.4** 功能详述 | 5 子节：定位 / UI / 控件交互 / 弹窗状态异常 | ✅ **PM 全写** — 含 12 项必填齐全 | 不参与 |
| **5.x.5 数据规范** | 接口契约 + 字段表 | ⚠️ **PM 写「需求层」字段约束**（业务字段名 / 业务约束 / 业务默认值 / 用户能看到的语义） | **研发写「实现层」**（具体 HTTP 路径精确格式 / Request schema / Response 错误格式 / 字段长度边界 / 数据库列名 / 索引） |
| **5.x.6 验收标准** | 可观察可判定 | ✅ **PM 全写** — 用户能观察到的现象（"< 500ms 显示 tooltip" / "primary blue + ←推荐") | 不参与 |
| **8.1 影响范围概览** | 改哪些模块 | ✅ **PM 全写** — 标记"涉及前端 / 后端 / 数据库" | 不参与 |
| **8.2 前端改动** | 文件清单 + 组件拆分 | ⚠️ **PM 只写「涉及哪些前端入口模块」**（如"InputBar 加 tooltip 子组件 + 提交预览折叠区"）| **研发写具体实现**（组件文件路径 / 类设计 / 状态管理方案 / 路由结构）|
| **8.3 后端改动** | 文件清单 + 接口 | ⚠️ **PM 只写「需要后端做什么」**（如"sanitize 接口 + audit_log 写入"）| **研发写具体实现**（API 路由 / Controller-Service 分层 / 事务边界 / 性能预算）|
| **8.3.1 错误码全集** | HTTP code + 错误码命名 + 触发场景 | ⚠️ **PM 只写「错误场景 + 用户看到的提示文案」**（如"风险词重复 → 用户看到'已存在该风险词'"）| **研发写具体错误码命名 + HTTP code**（如 `prompt_safety_dict_dup` / 400 / Response 格式）|
| **8.4 数据库改动** | 表 / 索引 / 迁移顺序 | ⚠️ **PM 只写「需要存什么字段」**（如"字典表存 risk_word / safe_replacement / category / source / created_by"）| **研发写表设计**（表名 / 字段类型 / 长度 / 索引 / 外键 / 迁移脚本）|
| **8.4.1 完整 DDL** | CREATE TABLE / ALTER / 回滚 | ❌ **PM 不写** — 这是研发工作 | **研发完整写** |
| **8.7 上线顺序** | 灰度策略 / 回滚条款 | ⚠️ **PM 只写「业务上线约束」**（如"先上后端再前端 / 灰度 10%/ 失败立刻回滚"）| **研发写实施方案**（具体灰度流量配置 / 监控指标 / 回滚命令 / 数据库迁移顺序）|
| **9. 待确认问题** | OPEN_QUESTIONS | ✅ **PM 全写** | 不参与 |
| **10-12. 上线指标 / 红线 / 工时** | 收尾 | ⚠️ **PM 写上线指标 + 红线 / 工时由研发评估** | **研发评估工时 / 性能红线** |

## 3. 边界口诀（写 PRD 时反复对照）

| PM 写 | 研发写 |
|---|---|
| **业务需要什么** | **怎么实现** |
| **字段对用户的语义**（"category = 风险词分类"）| **字段在数据库的存储**（"category VARCHAR(20) DEFAULT 'celebrity'"）|
| **验收能观察的现象**（"< 500ms 显示 tooltip"）| **性能预算的实现**（"前端 SWR 30s + 后端 Redis 60s"）|
| **错误用户看到的文案**（"⚠️ 字典中已有此风险词"）| **错误码命名**（`prompt_safety_dict_dup`）|
| **接口需要返回什么数据**（"返回字典条目列表 + 总数"）| **HTTP 方法 / 路径 / 状态码**（`GET /api/admin/prompt-dictionary` / 200 OK）|
| **流程图 Mermaid**（业务步骤）| **时序图 / 类图**（实现细节）|

```markdown
### 8.3.1 错误场景 + 用户提示文案（PM 草稿，研发深化具体错误码）
```

### PM 写 8.4 数据库需求草稿（业务视角）

```markdown
### 8.4 数据库改动需求（PM 草稿，研发设计表结构）

需要新建 / 改造的存储：

| 表名（业务名）| 存什么 | 业务约束 |
```

```markdown

❌ **本节由研发在技术 spec 中编写**，PRD 不含具体 CREATE TABLE 语句。
```

## 5. 实施时如何判断"我越界了吗"

PM 写到任何一段时**自问 3 个问题**：

1. **如果换一个不同技术栈的研发实施，这段还成立吗？**
   - ✅ 成立 = PM 该写（业务层）
   - ❌ 不成立（依赖特定技术栈如 PostgreSQL / Next.js）= PM 越界

2. **这段内容研发能 100% 不动地复制到代码 / DDL 里吗？**
   - ❌ 不能（需要研发判断）= PM 该写
   - ✅ 能（直接代码级）= PM 越界

3. **这段内容描述的是"用户能看到 / 操作 / 影响什么"还是"系统内部怎么处理"？**
   - 用户能看到 = PM 该写
   - 系统内部 = PM 越界

任一问题答 "越界" → 改成草稿 + 标 ⚠️ "研发深化"。

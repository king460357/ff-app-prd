# 输出格式标准

> 让 PRD 适合飞书 / Lark / 企业文档直接复制粘贴。

---

## 1. 代码块使用规则

### ✅ 允许使用代码块的内容

| 类型 | 示例 |
|------|------|
| Mermaid 流程图 | 用户操作流程、状态机 |
| ASCII 界面示意图 | 页面布局线框图 |
| 路由 / 接口 / 任务名 / 字段名 | `GET /api/v1/users`、`user_status` |
| 命令 / 脚本 | `npm install`、`python script.py` |
| JSON / YAML | 配置示例、数据结构示例 |
| SQL | 查询示例 |

### ❌ 禁止使用代码块的内容

| 内容类型 | 错误做法 | 正确做法 |
|---------|---------|---------|
| 功能说明 | 把整段功能描述放进代码块 | 用普通正文 |
| 交互说明 | 把"点击后弹窗"放进代码块 | 用普通正文 + 表格 |
| 按钮行为 | 用代码块描述按钮逻辑 | 用列表或表格 |
| 状态解释 | 用代码块描述每个状态 | 用表格 |
| 异常处理 | 用代码块写处理流程 | 用表格 |
| 验收标准 | 用代码块列验收点 | 用列表 |
| 修改原因 | 用代码块解释为什么改 | 用普通正文 |
| 产品注解 | 用代码块加注释 | 用普通正文或引用块（>） |

### 反例

❌ 错误：

````markdown
```
功能说明：
本功能用于让用户上传头像。点击上传按钮后弹出文件选择器。
```
````

✅ 正确：

```markdown
**功能说明**

本功能用于让用户上传头像。点击上传按钮后弹出文件选择器。
```

## 2. 正式 PRD 章节格式

### 推荐结构（PRD 必含 7 大类全覆盖）

```
1. 产品概述                    ← 必含：背景 & 目标
2. 目标用户与使用场景          ← 必含：用户场景
3. 核心用户动线（Mermaid）
4. 功能清单                    ← 必含：功能描述（总览）
   4.1 关键页面线框图（ASCII）  ← 必含：效果图（结构层）
   4.2 UI 设计图英文提示词      ← 必含：效果图（视觉层，HTML 实物 UI 渲染用）
5. 功能详细描述                ← 必含：功能描述（细节）+ 交互说明 + 验收标准
   5.1 [功能 A]
       5.1.1 功能说明
       5.1.2 页面位置
       5.1.3 页面结构
       5.1.4 UI 示意（ASCII）
       5.1.5 交互说明
       5.1.6 状态说明
       5.1.7 异常与置灰
       5.1.8 联动规则
       5.1.9 数据规范
       5.1.10 验收标准
   5.2 [功能 B]
       ...
6. 文案规范
7. 非功能性需求
8. 影响范围矩阵                ← 必含：影响范围（前端/后端/数据库/第三方/上线顺序）
   8.1 模块 × 端 总览
   8.2 前端影响详情
   8.3 后端影响详情
   8.4 数据库影响详情
   8.5 第三方依赖
   8.6 模块间数据流依赖（Mermaid）
   8.7 上线依赖顺序
9. 技术选型建议（如已有选型则跳过）
10. 待确认问题
```

## 3. 去重审查输出格式

模式 B 发现重复时，必须使用三段式（详见 `deduplication_rules.md`）：

```markdown
### 重复点 N：{重复主题}

**原文：**
> {复制用户文档中的原句或原段落}

**修改后：**
> {给出可直接替换的版本}

**修改原因：**
> {说明为什么重复、为什么这样改}
```

---

## 4. 完整输出要求

当用户要求"完整需求 / 可以直接复制 / 不要缩略"时，**禁止**使用以下表达：

| 禁止 | 替代 |
|------|------|
| 原文保留 | 完整重写 |
| 上文同理 | 完整展开 |
| 参考前文 | 重新展开（必要时加引用："详见第 X 章 X.Y 节"） |
| 表格略 | 完整复制原表 |
| 此处省略 | 标 `[待补充]` 并说明缺什么 |
| 你原来的内容不变 | 完整粘贴用户原内容 |

⚠️ 用户复制粘贴你的输出后，应该能直接用，不需要再回去看上文。

## 5. 表格使用规范

### 5.1 何时用表格

- 多项对比（功能对比、方案对比）
- 状态说明（状态 + 表现 + 触发条件）
- 字段说明（字段名 + 类型 + 必填 + 校验）
- 异常处理（场景 + 用户感知 + 系统响应 + 提示文案）

### 5.2 表格语法

优先使用 Markdown 表格（飞书/Lark 直接复制可用）：

```markdown
| 字段 | 说明 |
|------|------|
| 状态 | 使用中文状态 |
| 操作 | 明确按钮行为 |
| 异常 | 明确提示文案 |
```

### 5.3 表格内不要套代码块

❌ 错误：在表格单元格里嵌入多行代码块
✅ 正确：把代码块单独列出，表格里写"详见第 X 节"

## 6. 状态命名规范

正文中**必须使用中文状态**（详见 `chinese_term_standard.md`）：

| 推荐中文 | 错误示例 |
|---------|---------|
| 未开始 | idle、未触发 |
| 生成中 | running、进行中（也可用，但全文统一） |
| 已完成 | succeeded、完成（统一为"已完成"） |
| 生成失败 | failed、错误（统一为"失败"或"生成失败"） |
| 部分失败 | partial_failed、半成功 |
| 已保存 | saved |
| 未保存 | unsaved、未存 |
| 只读 | readonly |
| 已锁定 | locked |
| 可点击 | enabled、能点 |
| 置灰 | disabled、灰色 |

字段表（接口字段、数据库字段）可以保留英文枚举，但**必须有中文说明列**。

## 7. 飞书/Lark 复制兼容性

为保证复制到飞书/Lark 后格式不丢失：

1. ✅ Markdown 表格 → 飞书表格识别正常
2. ✅ Mermaid 代码块 → 飞书有 Mermaid 渲染插件
3. ✅ 多级标题（# ## ###） → 飞书自动转为大纲
4. ✅ 列表（- / 1.） → 飞书识别正常
5. ⚠️ HTML 标签 → 飞书可能识别异常，避免使用
6. ⚠️ 特殊字符（emoji 大量使用） → 飞书可能显示异常

---

## 8. 语气与内容密度

- ❌ 不要泛泛而谈："建议优化"、"可以考虑"、"可能需要"
- ✅ 给出可替换内容："改为：xxx"
- ✅ 保持丰富，但**不重复同一规则**（详见 `deduplication_rules.md`）
- ✅ 优先使用清晰表格 + 短段落
- ❌ 不使用花哨图标装饰文档

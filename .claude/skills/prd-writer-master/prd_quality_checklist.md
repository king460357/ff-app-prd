# PRD 质量检查清单

> 本文件用于模式 B（评估改进）。逐项对照检查，每项给出 ✅ / ❌ / ⚠️。

---

## 1. 完整性检查

每个核心功能是否包含：

- [ ] 功能目标
- [ ] 使用场景
- [ ] 入口位置
- [ ] 页面结构
- [ ] UI 示意图（ASCII 线框）
- [ ] 按钮交互（每个按钮 6 项必填）
- [ ] 弹窗规则（每个弹窗 5 项必填）
- [ ] 状态说明（6 类状态）
- [ ] 异常处理
- [ ] 联动规则
- [ ] 数据规范（字段名/类型/长度/必填/默认值/校验规则）
- [ ] 验收标准

---

## 2. 可开发性检查（研发视角）

研发看完后应能回答：

1. 从哪里进入？
2. 点哪个按钮？
3. 点击后发生什么？
4. 是否有弹窗？
5. 哪些状态？
6. 失败怎么办？
7. 是否需要刷新/同步？
8. 是否需要权限判断？
9. 是否有默认值？
10. 是否有边界条件？

只要有一项回答不了，说明 PRD 不达标。

---

## 3. UI 检查（设计师视角）

UI 看完后应能知道：

1. 页面整体结构
2. 模块上下位置
3. 列表 / 卡片 / 画布 / 弹窗样式
4. 按钮位置
5. 置灰状态
6. 加载中（loading）状态
7. 空状态
8. 错误状态

---

## 4. QA 检查（测试视角）

QA 看完后应能写出至少 8 类测试用例：

1. 正常流程
2. 失败流程
3. 部分成功流程
4. 权限不足流程
5. 状态变化流程
6. 重试流程
7. 数据同步流程
8. 页面刷新流程

---

## 5. 去重检查

检查同一规则是否出现在多个地方：

- 规则章节
- 场景章节
- UI 章节
- 流程图章节
- 测试章节
- 验收章节

如果出现重复，应保留唯一规则源，其他位置改为引用（"详见第 X 章 X.Y 节"）。

详见 `deduplication_rules.md`。

---

## 6. 语言与格式检查

- [ ] 是否有英文术语无中文说明？
- [ ] 是否有代码块滥用（把普通说明放进了代码块）？
- [ ] 是否有"原文保留 / 此处略 / 参考上文"等占位语？
- [ ] 是否有"可能 / 应该 / 大概 / 暂时无法"等模糊词？
- [ ] 状态命名是否使用中文？
- [ ] 是否过度口语化？
- [ ] 验收标准是否含"流畅 / 美观 / 友好 / 顺畅 / 清晰 / 高效 / 良好"等无法量化的模糊词？（应改为具体可观察指标如「< 500ms 显示」/「primary blue + ←推荐」）

详见 `output_format_standards.md` 和 `chinese_term_standard.md`。

## 7. PRD 控件完整性检查（2026-05-11 新增）

ASCII 草图里画的**每一个**可交互元素都必须在 5.x.3 控件清单里有对应 6 项必填描述。逐一核对：

- [ ] ASCII 草图里所有 `[xxx]` 按钮（含表格行内 [编辑] / [禁用] / [撤回]）都在 5.x.3 控件清单中？
- [ ] ASCII 草图里所有 `[xxx ▼]` 下拉（来源 / 类别 / 状态 / 时间范围等）都在 5.x.3 控件清单中？
- [ ] ASCII 草图里所有 `[搜索框]` / textarea / Input / Switch / Radio 都在 5.x.3 控件清单中？
- [ ] ASCII 草图里所有 `◉/○` 单选项 / `▼/▶` 折叠箭头 / Tab 切换器都在 5.x.3 控件清单中？

**常见漏列**：顶部 toolbar 的搜索框 + 筛选下拉、输入区的 textarea 本身、折叠区的展开收起、Tab 切换器。

**自检方法**：写完 5.x 后逐一列出 ASCII 草图里的所有控件标记，对照 5.x.3 控件清单——漏一个就视为 PRD 不完整需返工。

## 8. PRD 内部一致性检查（2026-05-11 新增）

同一字段名（如 `category` / `source` / `status` / `action`）的 enum / 类型 / 默认值在 PRD 多处必须**完全一致**：

- [ ] 5.x.4 弹窗字段表的 enum 与 5.x.5 schema 字段表 enum 一致？
- [ ] 5.x.5 schema 字段表 enum 与 8.4.1 DDL 注释一致？
- [ ] ASCII 草图里的提示（如 `↑ celebrity/music/...`）与字段表 enum 一致？
- [ ] v0 mockup 实现（如有）与 PRD 字段表 enum 一致？
- [ ] 字段类型（input vs textarea）在多处一致？
- [ ] 默认值在多处一致？

**自检方法**：写完 PRD 后 `grep -n "{字段名}" *.md` 列出所有出现位置，对照 enum / 默认值 / 字段类型。

**v0 ↔ PRD 冲突原则**：v0 mockup 与 PRD 文档冲突时，AI **不自决**——加 ⚠️ 块列出 4 处差异（v0 实际 / PRD 字段表 / schema / DDL）+ 建议方案 + 等用户拍板。

---

## 9. HTML 派生展示版质量检查（2026-05-11 新增）

每份 PRD 必有 `.html` 派生展示版。HTML 端核查：

- [ ] `grep -c '<pre><code>┌' file.html` = 0（无 ASCII Dialog 框）
- [ ] `grep -c '<pre><code>│' file.html` = 0（无 ASCII 边框字符）
- [ ] 所有 Dialog / AlertDialog / Popover 都用 `.dialog-mockup` 真实 HTML/CSS 渲染（不允许 `<pre><code>┌──┐│└──┘`）
- [ ] Calendar / 单选 radio / Switch / Tab 都用真实 HTML/CSS 渲染（不允许 ASCII）
- [ ] `grep -c 'data-jump=' file.html` ≥ ASCII 草图中按钮数 + 表格操作列按钮数（每个 mockup 元素都有 data-jump）
- [ ] `grep 'querySelectorAll.*data-jump' file.html` 命中（JS handler 存在）
- [ ] `grep '@keyframes jump-flash' file.html` 命中（动画 CSS 存在）
- [ ] `grep '\.jump-highlight' file.html` 命中（高亮类存在）
- [ ] HTML 顶部含 `<div class="notice notice-info">` "怎么打开看" 段
- [ ] dark theme 三锁色值正确（`#111111` / `#1F2937` / `#F9FAFB` / `#9CA3AF` / `#374151` / `#3B82F6`）
- [ ] 同目录有 `README.md` 引导本地打开（GitHub 自动渲染在目录列表底部）

**浏览器验证**：
- [ ] hover 任一 mockup 按钮显示蓝色 tooltip 「→ 见 5.x.y ...」
- [ ] 点击任一 mockup 按钮平滑滚动到对应章节 + 1.4s 蓝色闪烁高亮
- [ ] 5.5 三变体（如有）Tab 可切换
- [ ] 折叠区可展开收起

完整模板见 `html_ui_rendering_standard.md`。

## 10. PM 边界检查（2026-05-11 新增 — 不越界写技术 Spec）

PRD 是业务真理源，不是技术实现手册。逐项核查 PM 是否越界写到了研发该写的内容：

- [ ] 8.3.1 是否只写"错误场景 + 用户提示文案"（如「字典已存在该风险词」），**没有写**具体错误码命名（如 `prompt_safety_dict_dup`）？
- [ ] 8.4 是否只写"业务字段需求"（如「需要存 risk_word / safe_replacement / category」），**没有写**表设计 / 字段类型 / 长度 / 索引细节？
- [ ] 8.4.1 是否标注"由研发在技术 spec 中编写"，**没有写**完整 CREATE TABLE 语句？
- [ ] 5.x.5 数据规范是否只写"业务字段约束"（业务名 / 业务约束 / 用户语义），**没有写**精确 HTTP 路径格式 / Request Schema / 数据库列名 / 索引？
- [ ] 8.2 / 8.3 是否只写"涉及哪些前端 / 后端模块"，**没有写**具体类设计 / Controller-Service 分层 / 状态管理方案？
- [ ] 8.7 上线顺序是否只写"业务上线约束"（如"先后端再前端"），**没有写**灰度流量配置 / 具体监控指标 / 回滚命令？

**判断越界 3 问**（写每段时自问）：
1. **换不同技术栈研发实施，这段还成立吗？**（成立 = PM 该写；不成立 = PM 越界）
2. **研发能 100% 不动地复制到代码 / DDL 里吗？**（不能 = PM 该写；能 = PM 越界）
3. **描述的是"用户能看到 / 操作"还是"系统内部怎么处理"？**（用户能看到 = PM 该写；系统内部 = PM 越界）

任一答"越界" → 改成 PM 草稿 + 标 ⚠️ "研发深化"。完整边界规则见 `prd_vs_techspec_boundary.md`。

## 11. 多角色可读性检查

把 PRD 想象成 5 个角色在读，每个角色都应能从中获取自己需要的信息：

| 角色 | 关心什么 |
|------|---------|
| 产品经理 | 方向、决策依据、优先级 |
| 研发工程师 | 字段、状态、接口、逻辑 |
| UI/UX 设计师 | 页面结构、状态、交互 |
| QA 测试 | 验收标准、边界条件、异常处理 |
| 运营 | 用户路径、功能价值、文案 |

如果某个角色看完后还是一头雾水，说明 PRD 不够全面。

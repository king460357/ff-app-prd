# 团队工作流程（产品经理日常 + 6 岗位配合 + 跨平台操作手册）

> 这份文档解释我们这个 AI 协作仓库怎么用：**产品经理**每天怎么走、**前端 / 后端 / 数据库 / QA / UI** 怎么配合、信息怎么流转、遇阻怎么办。
> 新同事第一份必读，老同事查 FAQ。

---

## 一、整体协作图（10 秒看懂）

```
┌──────────────────────────────────────────────────────────────────┐
│  产品经理（PM）：用户反馈 → 决策表 → PRD → UI 提示词 → 落盘到仓库 │
└──────────┬───────────────────────────────────────────────────────┘
           │ 一次性同步给所有岗位（拉 Git）
           ▼
┌──────────────────────────────────────────────────────────────────┐
│  6 岗位 AI 拿到全套文件后并行 / 串行干活：                        │
│                                                                   │
│   UI       ──→  数据库  ──→  后端  ──→  前端  ──→  QA            │
│      ↓             ↓           ↓          ↓         ↓             │
│   出 UI 稿     出 DDL    出接口契约   出代码    出测试方案 + 执行 │
│                                                                   │
│   遇到 PRD 没覆盖的问题 → 写入 OPEN_QUESTIONS.md（异步）          │
└──────────┬───────────────────────────────────────────────────────┘
           │
           ▼
┌──────────────────────────────────────────────────────────────────┐
│  产品经理按自己节奏批量看 OPEN_QUESTIONS（不阻塞干活）             │
│  默认无问题 = 自动通过；有问题 → 用户裁定 → 同步答案              │
└──────────────────────────────────────────────────────────────────┘
```

**核心原则**：产品经理只对"决策本身"负责；客观信息（PRD / 接口 / DDL / UI）所有人**自助查仓库**，不需要找 PM 问。

## 二、新同事 5 分钟上手

### 2.1 准备环境

| 项 | macOS | Windows |
|---|---|---|
| 必装 | Cursor 编辑器（含 Claude Code 插件 / API Key 配置）| 同左 |
| Git 客户端 | 系统自带 / Homebrew 装 git | Git for Windows（含 Bash）|
| 终端 | 默认 Terminal / iTerm2 | **PowerShell 7+ 推荐**（不要用 cmd）|
| 编码 | 默认 UTF-8 ✅ | **首次打开 PowerShell 后执行**：`chcp 65001` 切换到 UTF-8，避免中文路径乱码 |

### 2.2 克隆仓库 + 启动

```bash
git clone <团队 Git 仓库地址>
cd product
cursor .                 # 用 Cursor 打开
```

在 Cursor 中按 `Cmd + L`（mac）或 `Ctrl + L`（Windows）打开 Claude 对话面板。Claude 会**自动读取**仓库根目录的 `CLAUDE.md`（项目总规则），无需手动告知。

### 2.3 告诉 Claude 你的角色

打开对话后第一句话：

> "你好。我是 **{你的角色}**（前端 / 后端 / 数据库 / QA / UI / 产品经理 任选一），需要在这个仓库做事。请先读 `CLAUDE.md` 和当期需求目录的主 PRD frontmatter 中的 `ai_context` 角色路由，然后告诉我我应该读哪些文件、输出哪些文件、跳过哪些文件。"

### 2.4 5 分钟上手 Checklist

- [ ] 1. 安装 Cursor + 配置 Claude Code 插件 / API Key
- [ ] 2. `git clone` 仓库到本地
- [ ] 3. Windows 用户在 PowerShell 执行 `chcp 65001`
- [ ] 4. 用 Cursor 打开仓库根目录
- [ ] 5. 启动 Claude（`Cmd + L` / `Ctrl + L`），它自动读 `CLAUDE.md`
- [ ] 6. 告诉 Claude："我是 {你的角色}，请告诉我读哪些文件、输出哪些文件、跳过哪些文件"
- [ ] 7. Claude 引导你读本文件第四章自己的角色行
- [ ] 8. 进入当期需求目录 `docs/{YYYYMMDD}/PRD/{需求}/`，第一眼读主 PRD frontmatter 的 `ai_context` 角色路由
- [ ] 9. 按主 PRD frontmatter 中你这个角色对应的章节清单读完该读的内容
- [ ] 10. 开始干活；遇阻写 `OPEN_QUESTIONS.md`

## 三、产品经理每天工作流程（5 个典型场景）

### 场景 1：收到本期反馈，要做决策

> 「每期 1 份汇总文档」流程：产品总监按周/期把当期反馈整理成 1 份 xlsx / md / 飞书表格导出 / 截图给 agent，agent 直接从中提取归类，不再按条粘贴单条 md。

| # | 操作 | 读什么 | 做什么 | 输出到 |
|---|---|---|---|---|
| 1 | 把本期反馈汇总文档给 Claude | — | 对话："本期反馈在 {文件路径或 URL}，帮我做决策" | — |
| 2 | Claude 调用 `demand-decision-agent` | 用户提供的反馈源（xlsx / md / 飞书等）| 直接提取并归类（不再按条拆 md） | — |
| 3 | 反馈原文归档到本批次 | — | Claude 把反馈文件拷到 `docs/{YYYYMMDD}/feedback/`（与本批次 PRD / decision-making 同级） | `docs/{YYYYMMDD}/feedback/{反馈文件名}` |
| 4 | 进入三轮决策 | Claude 自动读 `confirmed-decisions.md` + `decision-priority-queue.md` + 反馈原文 | 按"反馈决策按类不按案例"归到 N 类，每类 3 个候选方案 + 推荐 | — |
| 5 | 三轮决策完成 | — | 确认最终方案 | `docs/{YYYYMMDD}/decision-making/_汇总/{YYYYMMDD}_需求决策汇总.md` 自动落盘 |
| 6 | 决策同步到已确认决策 | — | Claude 自动追加到 confirmed-decisions.md | `docs/confirmed-decisions.md`（含顶部速查表更新）|

**耗时**：本期 N 类决策合计 30-60 分钟（按类不按案例）。

> 💡 **场景 1 用到的 agent / skill**：`demand-decision-agent`（自动归档 + 三轮决策）+ `Demand Solution Decision` skill（被 agent 内部调用做方案对比）。

### 场景 2：决策完成后，要写 PRD（双产物：md + html）

| # | 操作 | 读什么 | 做什么 | 输出到 |
|---|---|---|---|---|
| 1 | 告诉 Claude 写 PRD | — | 对话："基于刚才的决策表写 PRD" | — |
| 2 | Claude 调用 `prd-writer-agent` | 自动读决策表 frontmatter（继承需求人 6 字段，无需 `[待填]`） | 等 Claude 产出主 PRD（自包含 5 类角色路由 / 决策上下文 / UI 提示词 / DDL / 错误码）+ 同名 HTML 派生产物 | 双产物（.md + .html）|
| 3 | **关键步骤：PRD 出门前隐含决策反向沉淀** | Claude 主动反问"我们对话里讨论了哪些放弃方案 / 边界 / 角色易踩坑？" | 你回答后 Claude 同步沉淀到主 PRD 第 1 章决策上下文段 + 1.2.1 决策溯源块 | 主 PRD |
| 4 | 落盘后双产物保持一致 | — | 用户提需求变化时 Claude 必须**同时改 .md 和 .html**两份，确保最终一致 | 主 PRD + HTML |

**耗时**：1-3 小时（视需求复杂度）。

**输出双产物**：

```
docs/{YYYYMMDD}/PRD/{需求名}/
├── {YYYYMMDD}_{需求名}_PRD.md     ← 真理源（含 frontmatter ai_context 5 角色路由 + 第 1 章决策上下文 + 第 4.3 节 UI 提示词 + 第 5 章每模块「目标读者」标签 + 第 8.3.1 错误码 + 第 8.4.1 DDL）
└── {YYYYMMDD}_{需求名}_PRD.html   ← 派生产物（dark theme + 实物 UI 渲染 + 顶部"怎么打开"段，事实严格忠于 markdown）
```

### 场景 3：UI 评审与展示（HTML 派生产物）

| # | 操作 | 读什么 | 做什么 | 输出到 |
|---|---|---|---|---|
| 1 | 打开当期需求目录的 `{YYYYMMDD}_{需求名}_PRD.html` | 该文件 | 浏览器双击或 `open` 命令打开 | — |
| 2 | 看 HTML 内嵌的实物 UI 区块 | dark theme + CSS 渲染（非 ASCII）| 老板 / 跨团队评审会直接看 demo；产品经理自审稿 | — |
| 3 | 需要调整 UI | — | 直接告诉 AI"5.1 卡片间距改成 12px"，AI 同时改 markdown + html 两份 | 双产物同步 |
| 4 | 前端拿 HTML 与主 PRD 对照实现 | — | 前端实现按主 PRD 第 5 章交互逻辑 + HTML 实物 UI 区块视觉作参考 | 前端代码 |

**耗时**：评审会展示 5-10 分钟；UI 调整每轮 5-10 分钟。

### 场景 4：开发端写代码遇到问题，找你答疑

| # | 操作 | 读什么 | 做什么 | 输出到 |
|---|---|---|---|---|
| 1 | 启动 Claude，**让它先用 open-questions-triage 分诊** | — | 对话："看下 OPEN_QUESTIONS"，Claude 自动调用 `open-questions-triage` skill | 分诊报告（按紧急度排序 + 回答候选 + FAQ 候选） |
| 2 | 看分诊报告，对每条 Q 选「采纳草稿 / 修改 / 否决」| 当期需求目录的 `OPEN_QUESTIONS.md` | 🔴 阻塞条目优先处理；其他批量裁定 | — |
| 3 | 用户确认后 skill 自动回写 OPEN_QUESTIONS.md | — | 每条 Q 状态行变为"用户已裁定 [日期]：[结论]"；阻塞标记自动移除 | `OPEN_QUESTIONS.md` 自动更新 |
| 4 | FAQ 候选（重复 ≥ 3 次）升级 | — | 把候选搬到主 PRD 第 1 章 FAQ 段 | 主 PRD |

**核心**：**你不是同步审核者**，开发端不会等你才能写代码。你是异步答疑者。

> 💡 **场景 4 用到的 skill**：`open-questions-triage`（分诊 + 起草回答候选，详见 `.claude/skills/open-questions-triage/SKILL.md`）。把每天答疑时间从 ~30 分钟压缩到 ~5 分钟。

### 场景 5：需求需要调整 / 大改

| # | 操作 | 输出 |
|---|---|---|
| 1 | 重新跑 `demand-decision-agent` 三轮决策 | 新版决策表（旧版 `mv` 进 `_archive/`）|
| 2 | 重新跑 `prd-writer-agent` 增量更新 PRD（模式 C） | 主 PRD 升版本号；旧版进 `_archive/00_PRD_主文档_{日期}.md`；新版含"决策溯源块"指明取代原因 |
| 3 | 同步更新 `confirmed-decisions.md` 顶部速查表 + 加新调整记录 | confirmed-decisions.md |

### 场景 6：研发上线节奏（归研发方负责）

> 上线节奏由研发方负责规划与追踪。产品方只做：需求决策 + PRD 落盘 + HTML UI 派生产物 + ai-review 对齐。研发拉到 prd-repo 的批次目录后，自行规划上线节奏 / 任务分配。
>
> 如果 PM 需要看上线进度，**直接问研发**或看研发那边的项目管理工具（不在本仓库）。

## 四、各岗位「读什么 / 输出什么 / 跳过什么」

### 4.1 启动顺序（串行依赖）

```
UI            ──┐
                 │
                 ├─→  数据库  ──→  后端  ──→  前端  ──→  QA
                 │
（产品经理）     ──┘
```

UI 与数据库可并行。数据库 → 后端 → 前端串行。QA 等所有人完成。

### 4.2 全岗位通用必读（不分角色）

| 文件 | 用途 | 频次 |
|---|---|---|
| `CLAUDE.md` | 项目总规则（输出格式 / 命名规范 / 落盘路径）| 第一次进入仓库时 |
| 当期需求目录的主 PRD frontmatter `ai_context` | 角色路由（每期都不同）| 每次进入新需求目录时 |

### 4.3 各角色详表（主 PRD 自包含 5 类角色路由）

> 主 PRD frontmatter `ai_context` 已含 5 类角色的章节路由；下表是对应角色的快速速查。

#### 前端 AI

| 项 | 内容 |
|---|---|
| 必读核心 | 主 PRD 第 1-6 章 + 第 8.2 节（前端影响详情）<br>主 PRD 第 1 章「前端易犯偏差」段<br>当期上游决策表 |
| 输出 | 前端组件代码 + 接口调用 |
| 必跳过 | 主 PRD 第 8.4 节详细 DDL（你只需知道有哪些表）|
| 上游依赖 | 等数据库 AI 的 DDL + 后端 AI 的接口契约落盘后才启动 |
| 下游 | QA AI 等你的代码后写测试 |

#### 后端 AI

| 项 | 内容 |
|---|---|
| 必读核心 | 主 PRD 第 1 / 3 / 5 / 7 章 + 第 8.3 节（含 8.3.1 错误码全集）<br>主 PRD 第 1 章「后端易犯偏差」段<br>当期上游决策表 |
| 输出 | 后端接口实现 + OpenAPI/Swagger 接口契约 |
| 必跳过 | 主 PRD 4.1 ASCII 线框图细节 |
| 上游依赖 | 等数据库 AI 的 DDL 落盘 |
| 下游 | 前端 AI 等你的接口契约 |

#### 数据库 AI（前置依赖，最先启动）

| 项 | 内容 |
|---|---|
| 必读核心 | 主 PRD 第 8.4 节（含 8.4.1 完整 DDL）+ 各 5.x.9 字段表<br>主 PRD 第 1 章「数据库易犯偏差」段 |
| 输出 | DDL 完整脚本（含建表 / 修改 / 索引 / 回滚）|
| 必跳过 | UI / 交互 / 文案章节（5.x.5 / 5.x.6）|
| 上游依赖 | 无，你是前置依赖 |
| 下游 | 后端 AI 必读你的 DDL |

#### QA AI

| 项 | 内容 |
|---|---|
| 必读核心 | 主 PRD 全文（重点第 5 章各 5.x.10 验收标准 + 第 8 章上线顺序）<br>主 PRD 第 1 章「QA 易犯偏差」段 |
| 输出 | QA 测试用例 |
| 必跳过 | 视觉验收（由产品总监做）|
| 上游依赖 | 等前端 / 后端 / 数据库代码落盘后再启动 |
| 下游 | 无，你是最后一棒 |

#### UI AI

| 项 | 内容 |
|---|---|
| 必读核心 | 主 PRD 第 4 章（线框图 + 视觉系统）<br>主 PRD 第 4.3 节 UI 提示词（每模块自带 ABSOLUTE 段）<br>主 PRD 第 1 章「UI 易犯偏差」段 |
| 输出 | HTML 派生产物中的实物 UI 区块（CSS 渲染）|
| 必跳过 | 主 PRD 第 5 章交互逻辑细节、第 7 / 8 章 |
| 上游依赖 | 无，并行启动 |
| 下游 | 前端 AI 参考 HTML 实物 UI 区块视觉对齐 |

#### 产品经理（非 AI 角色，但需要协调全场）

| 项 | 内容 |
|---|---|
| 必做 | 把本期反馈汇总文档给 Claude（agent 自动归档到 `docs/{YYYYMMDD}/feedback/`）<br>跑 `demand-decision-agent` 三轮决策 → 落决策汇总<br>跑 `prd-writer-agent` 写 PRD 双产物（含 PRD 出门前隐含决策反向沉淀）<br>研发 ai-review → 落 `docs/{YYYYMMDD}/ai-review/`<br>跑 sync 脚本同步到 prd-repo |
| 输出文件 | `docs/{YYYYMMDD}/decision-making/_汇总/{YYYYMMDD}_需求决策汇总.md`<br>`docs/{YYYYMMDD}/decision-making/{需求}/{YYYYMMDD}_{需求简称}.md` 单条决策<br>`docs/{YYYYMMDD}/PRD/{需求}/{YYYYMMDD}_{需求}_PRD.md` 主 PRD（真理源）<br>`docs/{YYYYMMDD}/PRD/{需求}/{YYYYMMDD}_{需求}_PRD.html` HTML 派生产物<br>`docs/{YYYYMMDD}/ai-review/{YYYYMMDD}_{批次说明}_ai-review_v{N}.md`<br>`docs/{YYYYMMDD}/release-redline/上线红线.md`（多需求批次）|
| 不做 | 同步审核各岗位的实现 / 测试方案（已废弃，详见 `confirmed-decisions.md` #3.1）<br>编辑各角色实现细节（除非他们写入 `OPEN_QUESTIONS.md` 升级到你）|

### 4.4 关键约定（4 条）

1. **主 PRD 自包含**：5 类角色所需信息都在主 PRD 内（含 frontmatter ai_context 路由），默认不再单独建配套文件
2. **不返回审核给 PM**：写完直接干活，下游岗位 AI 读对方的单向交付物
3. **遇到 PRD 没覆盖的问题** → 写 `OPEN_QUESTIONS.md`，**不阻塞**自己写代码（继续干非阻塞部分）
4. **紧急阻塞场景**：在 `OPEN_QUESTIONS.md` 条目顶部加 `🔴 阻塞` 标记，PM 看到优先处理

## 五、信息流：所有人看到的内容必须一致

### 5.1 信息一致性的根基：所有信息都在 Git 仓库里

任何角色 AI 看到的信息**必须来自仓库内文件**，**不**依赖：

- ❌ 不依赖任何角色单独的对话上下文（Claude 在不同电脑是独立会话）
- ❌ 不依赖团队群聊 / 微信 / 飞书消息（那些不在 Git 里）
- ❌ 不依赖产品经理临时口头说明（口头说明必须先沉淀到文件再转发）

**唯一例外**：用户和产品端 Claude 的对话过程中产生的隐含决策——由 `prd-writer-agent` 在 PRD 出门前**强制反向沉淀**到主 PRD 第 1 章「考虑过但放弃的设计」/ FAQ /「五类角色专属易犯偏差」段 + 1.2.1 决策溯源块。详见 `prd-writer-agent.md` 第十七.A 节。

### 5.2 三类信息的真理源

| 信息类型 | 真理源 | 维护人 | 各角色 AI 怎么用 |
|---|---|---|---|
| **决策思路 / 商业判断 / 优先级 / 跨模块协调** | 用户 + 产品端 Claude | 产品经理 | **不能自己回答**；写入 `OPEN_QUESTIONS.md` 异步等用户答疑 |
| **客观信息**（PRD / 决策上下文 / DDL / 接口 / UI 提示词） | 仓库内对应文件（默认在主 PRD 内）| 产品经理首次产出 + 各角色 AI 单向交付物补充 | **自助查阅**，不需要找人问 |
| **过程信息**（决策演进 / 已废弃方案 / 反馈来源 / 历次变更） | 主 PRD 第 1 章决策上下文段 + `confirmed-decisions.md` 调整记录 | 产品经理（由 prd-writer-agent 同步沉淀）| **自助查阅**，前因后果都能追溯 |

### 5.3 任意角色都能溯源「为什么这么定」

```
当期主 PRD 1.2.1 决策溯源块（≤ 100 字概要）
        ↓
当期主 PRD 第 1 章「决策演进时间线」段
        ↓
当期主 PRD 第 1 章「考虑过但放弃的设计」段
        ↓
docs/{YYYYMMDD}/decision-making/{需求}/ 决策表 frontmatter（决策日期 / 优先级 / 来源）
        ↓
docs/{YYYYMMDD}/decision-making/_汇总/{YYYYMMDD}_需求决策汇总.md
        ↓
docs/confirmed-decisions.md 顶部「当前真理源速查」+ 各「调整记录」
        ↓
docs/{YYYYMMDD}/feedback/{反馈文件名}（本批次反馈原档）
```

任何角色拿任意决策点（如"为什么不做内部审核后台"），沿这条链向上溯源，最终能追到原始用户反馈或团队决策会议。

### 5.4 决策思路从产品经理流向各角色 AI

```
产品经理与产品端 Claude 三轮决策
        ↓
demand-decision-agent 落决策表（含 frontmatter 6 字段需求人信息）
        ↓
prd-writer-agent 写主 PRD（自包含决策上下文 + UI 提示词 + frontmatter ai_context 路由）+ HTML 派生产物
        ↓ 出门前 prd-writer-agent 强制反向沉淀对话隐含决策（关键兜底）
        ↓
各角色 AI 拉取仓库 → 读自己的必读清单 → 直接干活
        ↓ 遇阻：写 OPEN_QUESTIONS.md
        ↓
产品经理异步看 OPEN_QUESTIONS（按自己节奏批量回答）
```

## 六、跨电脑 / 跨操作系统的协作约定

### 6.1 macOS / Windows 兼容性

| 场景 | macOS | Windows |
|---|---|---|
| 中文文件名 / 目录名 | 直接支持 ✅ | Cursor 支持 ✅；命令行需先 `chcp 65001` |
| 含空格目录 | 已修复 ✅ | 已修复 ✅（仓库已用连字符替代空格）|

| 行尾（CRLF / LF）| LF | Git 默认转换；仓库根有 `.gitattributes` 强制 LF |
| 路径长度 | 无限制 | 260 字符上限（仓库当前最深约 75 字符，安全）|

### 6.2 文件命名硬规则（每期需求目录内）

```
docs/{YYYYMMDD}/PRD/{需求名}/
├── {YYYYMMDD}_{需求名}_PRD.md      ← 主 PRD 真理源（必有）
├── {YYYYMMDD}_{需求名}_PRD.html    ← HTML 派生产物（必有）
└── _archive/                       ← 历史版本归档（自动维护）
```

主目录里**只允许存在一份"当前版"双产物**，被取代的旧版必须移入 `_archive/` 子目录。配套文件（决策上下文 / 角色意图书 / QA 计划等）默认不建——主 PRD 已自包含 5 类角色所需信息。详见 [`CLAUDE.md`](../CLAUDE.md)。

### 6.3 拉取最新仓库的标准流程

每次开始工作前：

```bash
git fetch
git pull --rebase           # 拉取最新主分支
```

变更检查：

1. 看当期主 PRD frontmatter 的 `last_updated` / `changelog` 字段是否有变化
2. 看 `docs/confirmed-decisions.md` 顶部"当前真理源速查"表是否变更

如发现重大变更（PRD 升大版本号 / 决策方案被取代），**先停下手中的工作**，对照新版本调整自己的输出。

### 7.1 必须找产品经理（写 OPEN_QUESTIONS.md）的 5 类问题

### 7.2 OPEN_QUESTIONS.md 标准格式

```markdown

## Q[编号] 🔴 阻塞 | [角色] | ...
（紧急阻塞条目顶部加 🔴 阻塞 标记，用户优先看）
```

### 7.3 各岗位与产品经理的协作节奏

```
你（角色 AI）                  产品经理（用户）
   ↓
读必读 → 干活                     做自己的事（写下一个需求决策 / 评审上一期）
   ↓

   ↓ （异步，不阻塞自己）          按自己节奏批量看 OPEN_QUESTIONS

   ↓
```

## 八、常见 FAQ

### Q1：我读哪份 PRD？目录里有 v1.0 / v2.0.1 怎么办？

进当期需求目录看到的就是当前版双产物 `.md` + `.html`（旧版已移入 `_archive/` 子目录）。主 PRD frontmatter 中 `version` 字段明确写了"当前真理源 = vX.X"。

### Q2：我怎么知道决策为什么这么定？

按本文件 5.3 节溯源链路从 PRD 1.2.1 决策溯源块开始查，最终能追到原始反馈表。

### Q3：我的角色没找到必读清单？

看本文件第四章「各岗位读什么 / 输出什么 / 跳过什么」。如果你的角色不在 5 类（前端 / 后端 / 数据库 / QA / UI）里，问产品经理是否需要新建角色指引。

### Q4：我发现 PRD 写错了 / 字段冲突 / 已废弃方案被重新引入了，怎么办？

写入 `OPEN_QUESTIONS.md`，**不要**自己改 PRD。PRD 是产品经理的真理源，改动需要先经决策表层 → confirmed-decisions.md 同步。

### Q5：上游角色的意图书还没落盘，我能先开始吗？

不能（除了 UI 和数据库 / PM 是并行起点）。其他岗位等上游意图书落盘。如果上游迟迟不落盘，写入 OPEN_QUESTIONS 升级到产品经理。

### Q6：我用的是 Cursor 不是 Claude Code CLI，本手册适用吗？

适用。Cursor 内 Claude 与 Claude Code CLI 共享相同的 `CLAUDE.md` 加载机制。Windows 用户首次打开 PowerShell 后执行 `chcp 65001` 可避免中文路径乱码。

### Q7：我看到一个 `[待填]` 占位字段，能改吗？

如果是 PRD frontmatter 的 `product_owner` / `requester_*` / `stakeholders` 等需求人信息字段——这些应该由 demand-decision-agent 在决策表第一轮强制收集后自动写入，不应该有 `[待填]`。如果看到了，写 OPEN_QUESTIONS 提醒产品经理回头补全决策表 frontmatter。

### Q8：我能直接运行 demand-decision-agent / prd-writer-agent 吗？

只有产品经理能。各开发岗位 AI 在 Cursor 里启动 Claude Code 后，自动加载本仓库的 `CLAUDE.md`，无需也不应调用这两个 agent（它们的工具与产品经理流程绑定）。如果你是产品经理，直接告诉 Claude "我要做需求决策" / "我要写 PRD"，Claude 会自动路由到对应 agent。

### Q9：我可以自己改主 PRD 吗？

不建议。主 PRD 是产品经理的真理源，改动需要先经决策表层 → confirmed-decisions.md 同步。各角色 AI 发现 PRD 有问题应该写入 `OPEN_QUESTIONS.md`，而不是直接改 PRD。

### Q10：我的 AI 把文件写到了桌面怎么办？

提醒 Claude："按 CLAUDE.md 文件落盘红线规则，把文件放到 docs 对应路径"。仓库的 `CLAUDE.md` 和两个 SKILL 都已禁止写桌面。

### Q11：本批次反馈原文（`docs/{YYYYMMDD}/feedback/`）能 Git 同步吗？

可以同步——sync 脚本会把它推到 prd-repo `{YYYYMMDD}/feedback/` 让研发也能看到原始反馈。产品总监给反馈源前请**手动脱敏**（隐藏真实姓名 / 联系方式，关联人用化名 / 角色代替）。如某份反馈仍含敏感信息无法脱敏，单独把该文件加入 `.gitignore`，但**不要把整个 `feedback/` 目录排除**——否则团队成员的 Claude 看不到上下文，会胡乱猜测背景。

## 九.5、Sync 安全机制（2026-05-12 起）

⚠️ **核心红线**：sync 脚本**默认 commit 后停，不自动 push**。要 push 必须显式确认。

### 为什么这样设计

云端 commit 一旦 push 上去，**审计日志和 git history 永远留痕**。误推 demo / 测试 / 在途稿件造成的影响：
- 跨团队协作者已经看到（无法回收）
- 重要决策文档历史被污染
- 强制 force push 删除会破坏其他人的本地分支

2026-05-12 之前 sync 默认自动 push，曾误推 demo 演示批次到 prd-repo，后续要专门清理。**改为"commit 后停 + 用户审查 + 显式 push"三步走杜绝误推**。

### PM 的 sync 工作流（3 步）

#### 第 1 步：sync（commit 到 staging，不 push）

```bash
# 给研发看本期需求
bash scripts/sync-to-prd-repo.sh --batch YYYYMMDD -m "feat: ..."

# 给其他 PM 同事看 PM 工具包升级
bash scripts/sync-to-pm-workflow.sh -m "feat: ..."
```

脚本跑完会显示：
```
═══ ✅ 已 commit 到 staging（未 push）═══

📋 已 commit 的内容：
   abc1234 feat: ...

🔍 你可以现在审查内容：
   cd ~/prd-repo-staging && git show HEAD --stat
   open ~/prd-repo-staging/YYYYMMDD/

✅ 确认无误后跑 push：
   cd ~/prd-repo-staging && git push origin main
```

#### 第 2 步：审查 commit 内容

```bash
# 看 commit 改了哪些文件
cd ~/prd-repo-staging && git show HEAD --stat

# 看具体改动 diff
cd ~/prd-repo-staging && git diff HEAD^ HEAD

# 浏览器打开看实际文件
open ~/prd-repo-staging/YYYYMMDD/
```

#### 第 3 步：选 A / B / C

| 选项 | 命令 | 用途 |
|---|---|---|
| **A. 审查 OK，push** | `cd ~/prd-repo-staging && git push origin main` | 推到云端 |
| **B. 需调整** | 改完后重新跑 sync 产生新 commit（覆盖刚才的）| 改改改后再 push |
| **C. 撤回 commit** | `git reset --soft HEAD^`（保留改动，回到未 commit 态）<br>`git reset --hard HEAD^`（丢改动 — 谨慎）| 整体撤回这次工作 |

### 跟 AI 助手协作的口令

当 PM 跟 AI 助手（Claude / Cursor）协作 sync 时：

| PM 口令 | AI 做什么 |
|---|---|
| "同步给研发" / "同步给 PM 同事" | 跑 sync → commit 到 staging → 停在审查阶段 → 等用户确认 |
| "推 / push" | 跑 `git push origin main` |
| "撤回 / 别推" | 跑 `git reset --soft HEAD^` |
| "**直接 push**"（罕见，必须显式说）| sync 加 `--push` flag 一步推送 |

AI 助手**绝不在用户没明确说"push"前自动推送**——这是工作流硬约束。

### 显式 --push 的使用场景

只有 PM **已经在终端看过 staging 内容** + **明确说"直接推"** 时才用 `--push`：

```bash
# 例：你 review 完知道改动 OK，直接推（不再二次审查）
bash scripts/sync-to-prd-repo.sh --batch 20260512 --push -m "feat: ..."
```

99% 场景**不要用 --push**，按 3 步走才安全。

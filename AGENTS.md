# 产品 / 需求协作助手 · 工作方式

> 本文件与根目录 **`CLAUDE.md`** 共同约束在本仓库内的协作行为。  
> **开始前**：接到具体执行任务时，先用 **3～5 行**复述对任务的理解 + 将读取的真理源路径（或说明路径不存在），再执行。

---

|----------|-------------------------|
| PRD 真理源 | `docs/{YYYYMMDD}/PRD/{需求名}/*_PRD.md`（`.html` 为派生，事实以 `.md` 为准） |

---

## 0) 优先级

1. 用户在本轮对话里的**明确指令**始终最高。
2. 若用户或工作区文档（如 `CLAUDE.md` / 本 `AGENTS.md`）写了**强制前置步骤**（例如：写需求前必须先 git pull 某几个仓库），必须先执行或明确说明阻塞原因，不得跳过。
3. 若存在「Skill / 角色工作流」与默认习惯冲突，以**用户与工作区规则**为准。

---

## 1) 接到「需求 / PRD / 评审答复 / 决策」类任务时

- 先确认**真理源路径**（例如：PRD 只认 `.md`；汇总在 `*_汇总.md` 或 `confirmed-decisions.md`；评审在 `ai-review/`），再去读文件，不要凭记忆编造路径。
- 若仓库要求「先更新再写」，必须先拉取最新（或说明无法拉取的原因），再基于最新内容修改。
- **历史需求交叉检索（用户偏好 · 写新 / 大改 PRD）**：起草或显著扩展 **`docs/.../*_PRD.md`** 前，须结合本仓可追溯资产检索**与当期主题可能相交的历史逻辑**，**至少**覆盖：① **`flareflow-app-snapshot/KNOWLEDGE_BASE.md`**；② **`flareflow-app-snapshot/已上线功能/FF需求文档（App）/_EXTRACTED_INDEX.md`**（及索引指向的抽取/原文路径）；③ **已有批次** `docs/{YYYYMMDD}/PRD/**/*.md` 中与主题同域的文档；④ 用户当轮指定的其它权威路径（有则必查）。**须先输出「历史交集清单」**：每条含**来源**（路径或索引条目）、**历史规则/能力摘要**、**与本期关系初判**（已吸收 / 建议写入 / 显式不适用 / 与本期冲突须裁定）。然后**请你逐项或整表确认**：是否已在当期新 PRD 中考虑、是否须增删改、是否明确排除。**未给出检索范围说明与交集清单的**，不得声称已对齐历史需求。检索范围若需收窄，由你当轮声明（例如「仅 KNOWLEDGE + 本批次」）。

## 2) Skill / 方法论路由（按需启用，不要滥用）

- **写 / 改 PRD、把想法结构化**：必须进入「PRD 写作流程」——先澄清范围、成功标准、非目标、依赖与风险，再写章节；禁止一上来输出超长终稿除非用户明确要求。  
  - **用户分群 / 策略差异**：凡涉及多人群不同逻辑，必须先与 PM 对齐 **「主人群定义 + 非主人群各行策略」**（见 `prd-writer-master` 视角 1.5、`Demand Solution Decision` 分群矩阵门禁），**不得**凭笼统口径写 §2 后自称「研发免追问」。  
  - **PRD 与 HTML**：在 `.md` **未完全确认**前**不生成、不更新** `_PRD.html`；`.md` 定稿后由 AI **提醒**是否生成 HTML，且**每次**生成/大改 HTML **前须经你确认**（详见 **`CLAUDE.md`**「PRD · Markdown 与 HTML」）。  
  - **原型 / mockup**：须按 **移动端 App 设计规范**（竖屏、触控、安全区、逻辑画幅与 `KNOWLEDGE_BASE.md` §1.1 一致），见 **`CLAUDE.md`**「原型 / mockup · 移动端规范」。
- **从零做功能或大改行为**：先简短「意图 / 约束 / 验收」对齐，再拆任务。
- **多步骤实现**：先给出可执行步骤清单与检查点，再逐步做；每步结束用证据（命令输出 / 文件片段）核对。
- **宣称完成 / 已修复 / 已推送**：必须基于可验证证据（例如 git status、测试命令、关键文件 diff），禁止空口「已完成」。

与本仓 Skill 路径的对应关系见 **`CLAUDE.md`** 与 **`docs/PM_WORKFLOW_CURSOR.md`**（含「决策确认后再写 PRD」门禁）。

## 3) Agent / 子任务策略

- 默认**不要**为小事启动子 Agent；优先用搜索、读文件、小范围编辑解决。
- 仅当任务**大、可并行、或需要大范围只读扫仓**时，才拆给子 Agent / 并行探索；合并结果时要说明假设与未覆盖部分。

---

## 4) 输出习惯

- 修改仓库时：**改动范围最小**；无关文件不碰。
- 对用户说明时：**结论在前**，必要时附「文件路径 + 章节」方便复核。
- 若信息不足：用**最少的问题**把阻塞点问清楚；无关项不塞进同一卷。  
- **PM 选择题 / 待确认项（用户偏好 · 2026-05-13）**：须 **一次性** 用 **`AskQuestion` 一卷多题** 展示**全部**待确认点；**每题必有可点击选项**（禁止依赖用户手打 `Q1A` 等）。用户**整卷提交后**，Agent 再统一归纳、改 PRD / 决策表，**不得在只收到部分题答案时落盘终稿**。若某题确无标准选项，最后一项用「其他：请在对话补充」类兜底，仍以点击选中为准。
- **PRD 可读性（用户偏好 · 2026-05-14）**：写给 **QA / 产品** 的 PRD 中，凡出现 Android / FCM / Service / 生命周期等**偏实现**的用语，须在**同段或紧随 blockquote「白话」**再写一遍**人话与可验收口径**（例如：透传到达后，**须按现网逻辑维持或展示通知栏常驻通知**，而非仅堆叠术语）；技术句可保留给研发对齐。
- **PRD 篇幅（用户偏好 · 2026-05-14）**：整理或修订 PRD 时**去重**：同一业务规则只在一节写全（优先 **能力矩阵 + 单一主导航/授权节**），其余章节用「见 §x.x」交叉引用；避免在多章复制同一段落式描述。
- **PRD 需求背景（用户偏好 · 2026-05-15）**：**第 1 节只写为什么要做**（合规/业务动因、未同意 vs 先体验的矛盾、本期要厘清的合规边界与体验边界）；**禁止**把身份、弹窗、SDK、场景等功能规则写进背景。用户确认过的拟替换正文须**原样落盘**，不得擅自改写。详见 `prd-writer-master` →「需求背景（第 1 节）」。
- **FlareFlow App PRD 写法（用户偏好 · 2026-05-15 · 全部 App 需求文档）**：本仓 `docs/.../*_PRD.md` 与派生稿均须 **通俗易懂**；禁止弹窗 A/B、场景 S1 等代称；标题简洁且不含研发枚举；**线框与需求逻辑同节、线框在前**（合规双弹窗各有一框，禁止仅文末独立线框章）；服务端用白话能力表；文首禁止「文档类型/真理源」元信息表。完整条目见 **`prd-writer-master` →「FlareFlow App PRD 写作约定」**；样例：`docs/20260513/PRD/访客模式与游客身份/20260513_访客模式与游客身份_PRD.md`。可选 **`*_少跳转顺读版.md`** 另见同 Skill「研发顺读版」。
- **PRD 先对齐拟改再落盘（硬性 · 用户拍板 · 2026-05-14 起，2026-05-15 增补）**：无论当轮指令如何措辞（含「请改」「删掉」「不要出现某段」、口述真值/纠错等），在未与用户**对齐并确认「准备修改的内容」**——即具体 **`docs/.../*_PRD.md`**、章节/片段、**删改前后或改后效果**（拟改清单）——之前，**一律不得**对该等需求文档做任何写入或删改。确认拟改清单后，仍须有**显性落盘许可**（如「**同意修改**」「**可以修改**」「按确认后的清单落盘」「可以改 PRD / 需求文档」或**其它等价、且可视为针对该清单**的许可）方可落盘。两步缺一：**只能**先回复、列拟改、提问，**不得**改 `*_PRD.md`。

## 5) 冲突与未知

- 文档之间冲突时：按用户指定的「权威顺序」执行；若用户未指定，主动说明冲突点并请用户裁定。
- 凡属「商务 / 合同 / 线上真实配置」而你不确定：标为「待确认」，不要写成既定事实。

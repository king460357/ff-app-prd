1. 需求背景

FlareFlow 是出海短剧 App，业务与数据会触达多国用户，须满足隐私与用户同意方面的合规要求（含公司上市筹备对「同意可追溯、未同意前数据最小化」的约束）。

在「用户还没明确同意 Privacy Policy / Terms」的阶段，产品仍希望用户能先体验核心内容（浏览首页、看免费剧），但不能按「已同意用户」的方式采集、上报或开通营销能力。

因此需要在本期把两件事说清楚并落到同一套产品规则里：

  • 合规边界：未同意前做到什么程度算合规、同意之后才能开放什么。
  • 体验边界：未同意时用户仍能用到哪些功能、哪些必须等同意或登录后再说。

3. 身份定义

用户侧 枚举 服务端有无「同意隐私」临时凭证 说明
访客 VISITOR 无 仅本地浏览；运营弹窗关断；不可把点赞/收藏/进度等写入服务端账号数据
游客 GUEST 有（游客会话凭证 guest_session_token） 未注册登录，但已同意政策；收藏、进度等与 现网游客 一致
已登录 LOGGED_IN 正式登录态（账号 token） 充值、会员等
游客会话凭证是什么：用户在 隐私弹窗 点「同意」，或在 隐私二级挽留弹窗 点「同意并继续」后，服务端记录隐私同意，并返回一串令牌给 App 保存；之后需要「记住用户行为」的接口（点赞、My List、进度等）须携带该令牌。过期或失效后须重新走同意流程（见 §10、§12）。

4.4 按钮合规

隐私二级挽留弹窗上「不同意」须与「同意并继续」同等可点，不得弱化——通过隐藏、置灰、缩小尺寸、加重操作步数等方式弱化「不同意」均禁止。颜色可主次区分（主操作橙色 + 次操作中性深色），但按钮尺寸、位置、字号、字重、可读性、点击区域应等同。

HTML 实现：两按钮使用相同高度/字号/字重/宽度/圆角/可点击区域；「同意并继续」使用 .btn-privacy-primary（baseline §3 CTA-A 橙色），「不同意」使用 .btn-privacy（中性深色）。

5. 触发场景与需求逻辑

两枚隐私弹窗的按钮、关窗、身份：一律按 §4.2、§4.3（先看线框再看表）。下表用 场景 列写清页面/入口，需求逻辑 列写完整行为。
场景触发条件需求逻辑
冷启 冷启，隐私未决策 进 App 即弹隐私弹窗。用户完成 §4.2 或 §4.3 任一合法按钮路径后：若变为 GUEST，关窗并在 Discover 正常浏览（对齐现网冷启同意后）；若仍为 VISITOR，关两枚隐私弹窗后访客浏览。埋点按现网冷启隐私流程。
底部 Tab（Reels / My List / Profile） VISITOR 点 Reels / My List / Profile（含经 Profile Tab 进设置） 弹隐私弹窗；同意前底部 Tab 不得切走，选中态保持 Discover。变 GUEST 后关窗并 切至用户所点 Tab 进入对应页；若在隐私弹窗点不同意、进入挽留弹窗后在挽留弹窗点「不同意」，关两枚弹窗后 底部仍 Discover。埋点按现网各 Tab 入口。
Discover 顶栏「退出访客模式」 VISITOR 点该入口 弹隐私弹窗（及可能出现的隐私二级挽留弹窗）。变 GUEST 后关窗，顶栏恢复非访客态（现网一致）；最终在挽留弹窗选「不同意」则关两枚弹窗、仍为 VISITOR。埋点按现网退出访客入口。
待解锁页 VISITOR 进入待解锁页（路由同现网） 进店即弹隐私弹窗（非进店后再点其它才弹）。变 GUEST 后关隐私弹窗，留在待解锁页，解锁/登录/会员按现网。若经两枚弹窗均在挽留弹窗点「不同意」：关两枚弹窗，默认 pop 回上一层；路由栈无上一层（如深链直开）时可留页但 禁止白屏/卡死。页内任何「同意隐私」入口须拉起 与全局相同的隐私弹窗，禁止仅用「请登录」代替勾选同意。埋点按现网待解锁页。
播放页部分点按 访客在播放页点 须先同意隐私的控件（范围见 §7） 每次点按弹隐私弹窗（可用半屏，仍为同一套两枚弹窗）。变 GUEST 后关窗，再点一次完成原操作；失败 Toast 见 §10。经两枚弹窗均在挽留弹窗点「不同意」则关两枚弹窗，留播放页且仍为 VISITOR；再次点同一控件须再弹隐私弹窗，直至 GUEST 或 LOGGED_IN。若点按会进待解锁/收银，仍须先走完两枚隐私弹窗流程；进入待解锁页后按上表「待解锁页」行执行。

6. 访客态弹层策略

6.1 允许弹出

  1. 隐私弹窗、隐私二级挽留弹窗
  2. 系统通知（推送）权限框及必要短说明（频次克制）；不含 ATT / 定位 / 通讯录等（另文拍板除外）

6.2 禁止自动弹出

在 访客身份路径（VISITOR）关断触发器，含全屏 / 半屏 / BottomSheet：

  • 登录引导（三方、付费后、到首页、切换账号、绑定领奖等）
  • 付费 / 转化 / 挽留（解锁、订阅、到期、各类退出挽留、广告挽留等）
  • 增长 / 运营（闪屏推荐、Tab 运营位等）
  • 好评引导
  • 非合规确认（如搜索历史一键清除）

播放页：黑名单适用于沉浸路由；运营层 不得遮挡 两枚隐私弹窗。
Toast 不计入弹窗黑名单。不得新增第三套隐私合规弹窗。

7. 播放页相关逻辑

适用 访客（未同意隐私；研发枚举 VISITOR）。游客、已登录按现网，不适用本节。
用户操作是否弹隐私弹窗
播放、暂停、返回（含等同行为的系统返回）否
上下滑换集；拖进度/音量/亮度（未点图标）；点空白仅唤出控制条；纯手势快进否
其余可点控件（选集、清晰度、倍速、字幕、点进度跳转、点赞、收藏、弹幕、任务、横竖屏、解锁等）是，每次点按
游客 / 已登录按现网，不适用本节

关窗后仍为 访客 时，再次点同一控件须 再次弹隐私弹窗，直至变为 游客 或 已登录。

12. 服务端能力

本节说明 服务端要提供哪些能力 才能支撑 §3～§8 的客户端行为。路径、方法名由研发在技术方案中映射现网或新增接口（本 PRD 不写具体接口 URL）；验收以 行为 为准。

12.1 同意隐私与游客凭证

能力何时发生服务端应做什么客户端结果
记录隐私同意并创建游客身份隐私弹窗「同意」，或挽留弹窗「同意并继续」持久化同意记录（版本、时间等按法务）；生成并返回 游客会话凭证（guest_session_token）身份变为 GUEST；后续写数据请求携带该凭证
续期凭证游客长期使用、凭证将过期在合法前提下延长或换发凭证，使用户无感用户仍可点赞、同步进度等
作废凭证用户撤回同意、风控、长时间未用、服务端策略等使旧凭证失效再调用写接口失败；可提示 §10「须重新同意隐私」并回到隐私弹窗流程
访客路径仍为 VISITOR不得为仅浏览 Discover 的用户创建上述凭证与「未同意隐私」一致；不得误当成游客写库

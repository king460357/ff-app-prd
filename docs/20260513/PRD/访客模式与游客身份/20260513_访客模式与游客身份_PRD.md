# FlareFlow · 访客模式与游客身份

## 1. 需求背景

FlareFlow 是出海短剧 App，业务与数据会触达多国用户，须满足隐私与用户同意方面的合规要求（含公司上市筹备对「同意可追溯、未同意前数据最小化」的约束）。

在「用户还没明确同意 Privacy Policy / Terms」的阶段，产品仍希望用户能先体验核心内容（浏览首页、看免费剧），但不能按「已同意用户」的方式采集、上报或开通营销能力。

因此需要在本期把两件事说清楚并落到同一套产品规则里：

- **合规边界**：未同意前做到什么程度算合规、同意之后才能开放什么。
- **体验边界**：未同意时用户仍能用到哪些功能、哪些必须等同意或登录后再说。

---

## 2. 需求概述

| 维度 | 规则 |
|------|------|
| 访客默认可用 | 底部 Tab 选中 **Discover**；可搜索、列表、按现网播免费集 |
| 须弹隐私弹窗的入口 | 冷启未决策；访客点 Reels / My List / Profile；顶栏「退出访客模式」；进入待解锁页；播放页部分点按（第 7 节） |
| 隐私合规弹窗 | 仅 **隐私弹窗** + **隐私二级挽留弹窗** 两枚；各入口 **同一套 UI** |
| 访客可弹其它 | **系统通知（推送）权限** 及必要说明（第 6 节白名单） |
| 充值 | **仅 `LOGGED_IN`** 可 IAP / Top-up |
| SDK | `VISITOR` 路径不 init 归因、分析、广告联盟（第 9 节） |
| 服务端写数据 | **访客**不得保存点赞、收藏、观看进度等到账号侧（无游客会话凭证）；**游客 / 已登录**按现网。失败提示见第 10 节 |

---

## 3. 身份定义

| 用户侧 | 枚举 | 服务端是否已有「同意隐私」的临时凭证 | 说明 |
|--------|------|--------------------------------------|------|
| 访客 | `VISITOR` | **无** | 仅本地浏览；运营弹窗关断；不可把点赞/收藏/进度等写入服务端账号数据 |
| 游客 | `GUEST` | **有**（游客会话凭证，研发字段 `guest_session_token`） | 未注册登录，但已同意政策；收藏、进度等与 **现网游客** 一致 |
| 已登录 | `LOGGED_IN` | 正式登录态（账号 token） | 充值、会员等 |

> **游客会话凭证是什么**：用户在 **隐私弹窗** 点「同意」，或在 **隐私二级挽留弹窗** 点「同意并继续」后，**服务端**记录隐私同意，并返回一串令牌给 App 保存；之后需要「记住用户行为」的接口（点赞、My List、进度等）须携带该令牌。过期或失效后须重新走同意流程（见第 10、12 节）。

## 4. 隐私弹窗

全 App 共用下表；**第 5 节** 按场景补充 **关窗后的页面 / Tab / 路由** 逻辑（弹窗按钮与身份变化不重复写）。

### 4.1 何时出现

| 弹窗 | 出现条件 |
|------|----------|
| **隐私弹窗**（主授权） | 第 5 节各触发条件满足时 |
| **隐私二级挽留弹窗** | **仅当**在隐私弹窗点击 **不同意**；可反复「隐私弹窗 → 隐私二级挽留弹窗」，直至在挽留弹窗上结束 |

出现二级挽留时，**隐私二级挽留弹窗叠在隐私弹窗之上**（或与之同时展示）；两枚弹窗在全 App **同一套 UI 组件**，视觉细节以 Figma 为准。

### 4.2 隐私弹窗

**线框（ASCII；全屏或底部大弹层均可；竖屏逻辑画幅 390×844 pt）**

```
┌─────────────────────────────────────────┐
│ Privacy & terms                         │
│ Please read and agree to our …          │  ← 短导语，`consent_lead`
│ Privacy Policy  ·  Terms of Service     │
│ [ ] I have read and agree to …          │  ← 法务要求时必选
│                                         │
│ [ Agree ]              [ Disagree ]     │
└─────────────────────────────────────────┘
  点 Disagree → 叠出隐私二级挽留弹窗；仍为访客 `VISITOR`，授权流程未结束
```

**需求逻辑 · UI 结构**

1. 短标题  
2. 导语 1～2 句（长披露进政策全文页，主弹窗禁止长列表）  
3. Privacy Policy、Terms of Service 链接  
4. 法务要求时：必选勾选一条；**禁止**用登录替代 consent  
5. 按钮：**同意** / **不同意**（与线框一致）

- 未读全文能否点同意：**置灰同意** 或 **点前二次提示** 二选一 → 远程配置 + 法务定稿。  
- 营销/推送与主条款分拆时：可加第二条可选勾选（默认不勾）；未分拆则不加。

**需求逻辑 · 按钮与身份**

| 操作 | 关窗 | 身份 |
|------|------|------|
| **同意**（勾选等满足法务） | 关隐私弹窗 | `GUEST`（服务端记录同意并下发游客会话凭证，见第 12.1） |
| **不同意** | 不关隐私弹窗，出隐私二级挽留弹窗 | 仍 `VISITOR` |

### 4.3 二次挽留弹窗

**线框（ASCII；与隐私弹窗成套；各入口同一界面）**

```
┌─────────────────────────────────────────┐
│ Continue in visitor mode?               │
│ If you don’t agree … visitor mode,      │
│ basic browsing only.                    │
│                                         │
│ [ Agree and continue ]    [ Disagree ]  │
└─────────────────────────────────────────┘
  点 Disagree → 关两枚隐私弹窗，访客浏览（进入访客模式）
  点 Agree and continue → 关两枚隐私弹窗，等同隐私弹窗「同意」→ 游客 `GUEST`
```

**需求逻辑 · UI 结构**

- 说明不同意则进入访客模式、能力受限。  
- 按钮：**同意并继续** / **不同意**（中文产品文案可为「不同意」；与线框一致）。  
- 法务若要求挽留弹窗内保留政策链/勾选：与隐私弹窗 **同等满足** 后才可点「同意并继续」。

**需求逻辑 · 按钮与身份**

| 操作 | 关窗 | 身份 |
|------|------|------|
| **不同意** | 关两枚隐私弹窗 | `VISITOR` |
| **同意并继续** | 关两枚隐私弹窗 | `GUEST`（与隐私弹窗「同意」同效：记录同意 + 下发凭证；**不再回到隐私弹窗**） |

### 4.4 按钮合规

隐私二级挽留弹窗上「不同意」须与「同意并继续」**同等可点**，不得弱化。

### 4.5 文案 Key

**隐私弹窗（`consent_*`）**

| Key | 英文骨架 |
|-----|----------|
| `consent_title` | Privacy & terms |
| `consent_lead` | Please read and agree to our Privacy Policy and Terms of Service to use the full app. |
| `consent_link_privacy` | Privacy Policy |
| `consent_link_terms` | Terms of Service |
| `consent_checkbox_required` | I have read and agree to the Privacy Policy and Terms of Service. |
| `consent_primary_cta` | Agree |
| `consent_decline_cta` | Disagree |

**隐私二级挽留弹窗（`visitor_confirm_*`；可映射旧 `decline_confirm_*`）**

| Key | 英文骨架 |
|-----|----------|
| `visitor_confirm_title` | Continue in visitor mode? |
| `visitor_confirm_body` | If you don’t agree to the Privacy Policy and Terms of Service, you’ll use FlareFlow in visitor mode with basic browsing only. |
| `visitor_confirm_primary_cta` | Agree and continue |
| `visitor_confirm_secondary_cta` | Disagree |

**其它 Key**：`discover_exit_visitor_cta`（Exit visitor mode）；`paid_gate_agree`（Agree to Privacy Policy，Guest 沿用现网）。

## 5. 触发场景与需求逻辑

> 两枚隐私弹窗的按钮、关窗、身份：**一律按第 4.2、4.3 节**（先看线框再看表）。下表用 **场景** 列写清页面/入口，**需求逻辑** 列写完整行为。

| 场景 | 触发条件 | 需求逻辑 |
|------|----------|----------|
| **冷启** | 冷启，隐私未决策 | 进 App 即弹隐私弹窗。用户完成第 4.2 或 4.3 节任一合法按钮路径后：若变为 `GUEST`，关窗并在 Discover 正常浏览（对齐现网冷启同意后）；若仍为 `VISITOR`，关两枚隐私弹窗后访客浏览。埋点按现网冷启隐私流程。 |
| **底部 Tab（Reels / My List / Profile）** | `VISITOR` 点 Reels / My List / Profile（含经 Profile Tab 进设置） | 弹隐私弹窗；**同意前底部 Tab 不得切走**，选中态保持 Discover。变 `GUEST` 后关窗并 **切至用户所点 Tab** 进入对应页；若在隐私弹窗点不同意、进入挽留弹窗后在挽留弹窗点「不同意」，关两枚弹窗后 **底部仍 Discover**。埋点按现网各 Tab 入口。 |
| **Discover 顶栏「退出访客模式」** | `VISITOR` 点该入口 | 弹隐私弹窗（及可能出现的隐私二级挽留弹窗）。变 `GUEST` 后关窗，顶栏恢复非访客态（现网一致）；最终在挽留弹窗选「不同意」则关两枚弹窗、仍为 `VISITOR`。埋点按现网退出访客入口。 |
| **待解锁页** | `VISITOR` 进入待解锁页（路由同现网） | **进店即弹隐私弹窗**（非进店后再点其它才弹）。变 `GUEST` 后关隐私弹窗，**留在待解锁页**，解锁/登录/会员按现网。若经两枚弹窗均在挽留弹窗点「不同意」：关两枚弹窗，**默认 pop 回上一层**；路由栈无上一层（如深链直开）时可留页但 **禁止白屏/卡死**。页内任何「同意隐私」入口须拉起 **与全局相同的隐私弹窗**，禁止仅用「请登录」代替勾选同意。埋点按现网待解锁页。 |
| **播放页部分点按** | **访客**在播放页点 **须先同意隐私的控件**（范围见第 7 节） | **每次点按**弹隐私弹窗（可用半屏，仍为同一套两枚弹窗）。变 `GUEST` 后关窗，**再点一次**完成原操作；失败 Toast 见第 10 节。经两枚弹窗均在挽留弹窗点「不同意」则关两枚弹窗，留播放页且仍为 `VISITOR`；再次点同一控件须再弹隐私弹窗，直至 `GUEST` 或 `LOGGED_IN`。若点按会进待解锁/收银，仍须先走完两枚隐私弹窗流程；进入待解锁页后按上表「待解锁页」行执行。 |

## 6. 访客态弹层策略

### 6.1 允许弹出

1. 隐私弹窗、隐私二级挽留弹窗  
2. 系统通知（推送）权限框及必要短说明（频次克制）；**不含** ATT / 定位 / 通讯录等（另文拍板除外）

### 6.2 禁止自动弹出

在 **访客**身份路径（`VISITOR`）关断触发器，含全屏 / 半屏 / BottomSheet：

- 登录引导（三方、付费后、到首页、切换账号、绑定领奖等）  
- 付费 / 转化 / 挽留（解锁、订阅、到期、各类退出挽留、广告挽留等）  
- 增长 / 运营（闪屏推荐、Tab 运营位等）  
- 好评引导  
- 非合规确认（如搜索历史一键清除）

**播放页**：黑名单适用于沉浸路由；运营层 **不得遮挡** 两枚隐私弹窗。  
**Toast** 不计入弹窗黑名单。不得新增第三套隐私合规弹窗。

## 7. 播放页相关逻辑

> 适用 **访客**（未同意隐私；研发枚举 `VISITOR`）。**游客**、**已登录**按现网，不适用本节。

| 用户操作 | 是否弹隐私弹窗 |
|----------|----------------|
| 播放、暂停、返回（含等同行为的系统返回） | 否 |
| 上下滑换集；拖进度/音量/亮度（未点图标）；点空白仅唤出控制条；纯手势快进 | 否 |
| 其余可点控件（选集、清晰度、倍速、字幕、点进度跳转、点赞、收藏、弹幕、任务、横竖屏、解锁等） | **是，每次点按** |
| 游客 / 已登录 | 按现网，不适用本节 |

关窗后仍为 **访客** 时，再次点同一控件须 **再次弹隐私弹窗**，直至变为 **游客** 或 **已登录**。

## 8. 各身份能力一览

| 能力 | 访客（`VISITOR`） | 游客（`GUEST`） | 已登录（`LOGGED_IN`） |
|------|---------|-------|-----------|
| Discover 浏览、搜索 | ✓ | ✓ | ✓ |
| Reels / My List / Profile 根页 | ✗（底部 Tab 场景：先隐私弹窗，同意前 Tab 不切） | ✓ | ✓ |
| 免费集正片 | ✓ | ✓ | ✓ |
| 付费集正片 | ✗；进待解锁页场景 | ✗ | 现网权益 |
| 点赞 / 收藏 / 进度保存到服务端 | ✗（无凭证，请求应拒绝或客户端不发起） | ✓（带游客会话凭证） | ✓（登录态） |
| Top-up / IAP | ✗ | ✗ | ✓ |
| 归因 / 分析 / 广告 SDK | ✗ | 依法务 | ✓ |
| 运营类阻断弹层 | 不自动弹 | 现网 | 现网 |

## 9. 访客态第三方 SDK

| 类型 | 要求 |
|------|------|
| 归因、行为分析、广告联盟、IAP | 不 init |
| Crash | 仅匿名（无 user id / IDFA / GAID） |

| 类别 | 要点 |
|------|------|
| 身份 / Tab | 访客无游客会话凭证；Discover 选中；底部 Tab / 退出访客场景：同意前不切 Tab |

## 12. 服务端能力

本节说明 **服务端要提供哪些能力** 才能支撑第 3～8 节的客户端行为。路径、方法名由研发在技术方案中映射现网或新增接口（**本 PRD 不写具体接口 URL**）；验收以 **行为** 为准。

### 12.1 同意隐私与游客凭证

| 能力 | 何时发生 | 服务端应做什么 | 客户端结果 |
|------|----------|----------------|------------|
| **记录隐私同意并创建游客身份** | 隐私弹窗「同意」，或隐私二级挽留弹窗「同意并继续」 | 持久化同意记录（版本、时间等按法务）；生成并返回 **游客会话凭证**（`guest_session_token`） | 身份变为 `GUEST`；后续写数据请求携带该凭证 |
| **续期凭证** | 游客长期使用、凭证将过期 | 在合法前提下延长或换发凭证，使用户无感 | 用户仍可点赞、同步进度等 |
| **作废凭证** | 用户撤回同意、风控、长时间未用、服务端策略等 | 使旧凭证失效 | 再调用写接口失败；可提示第 10 节「须重新同意隐私」并回到隐私弹窗流程 |
| **访客路径** | 仍为 `VISITOR` | **不得** 为仅浏览 Discover 的用户创建上述凭证 | 与「未同意隐私」一致；不得误当成游客写库 |

### 12.2 内容与播放

| 能力 | 何时发生 | 服务端应做什么 | 验收（白话） |
|------|----------|----------------|--------------|
| **能否播放某一集** | 起播、换集、进待解锁前后 | 按剧集付费/免费规则、用户身份（访客/游客/会员）返回 **可播 / 不可播 / 须解锁** | 访客可播免费集；付费集不可直接播正片，与「待解锁页」场景、第 8 节一致 |
| **Discover 列表与搜索** | 访客浏览首页 | 提供与现网一致的列表/搜索数据（不要求登录） | 访客可不弹窗浏览 Discover |

### 12.3 点赞收藏与进度

以下能力在 **`GUEST` / `LOGGED_IN`** 下按现网游客或登录用户处理；**`VISITOR` 不得成功写入**（客户端不应带凭证发起，服务端亦应拒绝）。

| 能力 | 典型产品行为 | `VISITOR` | `GUEST` / `LOGGED_IN` |
|------|--------------|-----------|------------------------|
| 点赞 / 取消赞 | 播放页、列表心形等 | 不写入；客户端可先弹隐私弹窗，同意后再点一次 | 写入，须带对应身份凭证 |
| 收藏 / My List | 加入或移除「我的列表」 | 同上 | 同上 |
| 观看进度 | 续播、进度条同步 | 同上 | 同上 |
| 弹幕等其它账号侧数据 | 若产品已开放给游客 | 同上 | 按现网 |

### 12.4 登录后数据合并

用户从 **游客** 注册/登录为 **正式账号** 时，若现网支持 **合并** 游客期间的收藏、进度等到账号下，行为 **保持不变**；本需求不新增合并规则，但不得因访客/游客分流而破坏既有合并。

### 12.5 接口说明

- 具体 REST / RPC **URL、字段名** 不在本 PRD 列出，避免与实现脱节。  
- 接口清单在技术方案或 API 文档维护。

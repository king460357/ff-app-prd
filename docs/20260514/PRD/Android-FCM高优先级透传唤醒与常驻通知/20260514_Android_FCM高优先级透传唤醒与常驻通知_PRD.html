FlareFlow · Android FCM 高优先级透传唤醒与常驻通知 — PRD

0. 文档怎么读（写给 QA / 产品）

  • 本需求没有新 App 页面；用户能看到的仍是系统通知栏里那条长期存在的 FlareFlow 常驻通知（短剧相关）。
  • 文中会出现 FCM、Data、Service 等实现用语：每一段技术说明下面，若紧跟 「白话」引用块，表示同一段话的人话版，验收时以白话 + 验收小节为主即可。
  • 核心链路（整份 PRD 只讲这一件事）：服务端发一条用户看不见的透传消息 → 手机尽可能把 App 唤醒一下 → App 里要走现成那套逻辑，保证通知栏里那条常驻通知还在（或重新出现），从而继续合法占用「前台服务」能力。不要再单独弹一条「营销推送」。

1. 产品概述

1.1 背景

FlareFlow Android 端已实现长期展示的常驻通知（与短剧业务绑定的前台服务形态，具体文案与行为以线上一致为准）。当前 Android 端部分推送能力依赖本地推送；为在进程被系统回收或处于不利后台状态后提高应用被拉起的概率，使既有常驻通知所依赖的前台能力尽快恢复，需要引入 FCM 数据消息（Data-only，透传） 作为不展示在前端的远程唤醒信号。

白话：用户下拉通知栏，始终能看到一条 FlareFlow 的常驻通知（现网如此）。有时 App 在后台会被系统「收起来」，这条通知对应的能力会不稳。我们希望服务器悄悄发一条不显示给用户的消息，让 App 有机会醒过来，然后继续把这条常驻通知按规矩显示好。

1.2 问题陈述

  1. 仅依赖本地机制时，进程长时间未活跃可能被系统限制，常驻通知对应的前台能力不稳定。
  2. 需要一种不向用户展示额外推送 UI（无 FCM notification payload、无 App 内弹层）的远程手段，尽力唤醒应用并走既有常驻通知链路。

1.3 目标

目标 可验收描述
透传形态 Android 端收到的唤醒消息为 FCM Data-only；不包含 FCM notification 字段，系统不因此自动插入一条「推送栏营销通知」。
优先级 该类消息在 FCM 发送侧 100% 使用 high priority（与对话拍板一致）。
客户端行为 FirebaseMessagingService 收到消息后轻量执行：确保与短剧常驻通知绑定的 Foreground Service / startForeground 逻辑处于产品定义的有效态（与现网常驻策略一致，见 §5.1）。
服务端频控 默认 相邻两条间隔 ≥ 30 分钟、单设备 ≤ 48 条 / 自然日、同一自然分钟内最多合并为 1 条；支持自适应退避（见 §5.2）。
诚实边界 文档与评审口径明确：不承诺「每次必醒、秒醒」；用户强制停止、部分 OEM 冻结等情形下可能长期收不到（见 §7）。

白话（QA）：透传 = 用户不会看到一条新来的推送广告；high priority = 服务端把消息标成「急件」尽量送到；客户端行为 = 收到这条看不见的消息之后，App 要做的事就是：按线上一贯做法，让通知栏里那条「一直在的」FlareFlow 常驻通知保持有效（不能靠这条消息再弹第二条骚扰通知）。

1.4 非目标

  • 不包含 iOS / APNs 对称能力（本需求 仅 Android）。
  • 不通过本需求新增「对用户可见的营销推送」或 App 内 Modal / Toast（除非现网另有独立需求）。
  • 不改变常驻通知的业务含义与合规叙事的权威来源：常驻文案、渠道分类、用户开关若已存在，以现网 + 法务结论为准；本需求只增加到达路径（FCM Data）与频控规则。
  • 不在本 PRD 中锁定具体 HTTP 路径、FCM Admin SDK 版本、服务端 Cron 框架名（见 §8.3 研发深化）。

1.5 已确认决策（对话拍板）

# 决策项 结论
D1 平台范围 仅 Android。
D2 业务绑定 常驻通知与 短剧 业务绑定；常驻长期展示（已实现）。
D3 用户可见性 FCM 仅 Data 透传；不在前端展示本推送意图（无系统通知样式、无 App 内展示）。
D4 业务目的 主要目的为 唤醒应用，使既有 常驻通知 / 前台服务 策略继续生效，延长可运行机会（尽力而为，见 §7）。
D5 FCM 优先级 必须 high priority。
D6 默认频控 间隔：默认相邻两条 ≥ 30 分钟；日上限：单设备 ≤ 48 条 / 日；合并：同一自然分钟 最多 1 条；退避：检测到进程已活跃可拉长间隔（§5.2）。
D7 与本地推送关系 保留现网本地推送能力；本需求为 增量，不替代本地推送业务场景。

1.6 与相关能力的关系

  • 常驻通知 / 前台服务：本需求不重新定义其 UI 与业务规则，仅定义 FCM 到达后的客户端处理与 服务端发送策略。
  • 本地推送：继续承担定时/本地策略；与本需求 并行存在。

2. 目标用户与使用场景

2.0 用户分群与策略矩阵（产品真理源）

本需求无分群差异化产品行为（不向用户展示功能入口）；分群仅用于 服务端是否纳入唤醒调度 与 风控，具体标签以数据/运营为准，研发通过远程配置读取。

分群（产品命名） 判定摘要 是否接收唤醒类 FCM 说明
主人群：Android 已安装且已注册 FCM Token 设备存在有效 registration_token；未被服务端策略排除 可（仍受频控） 默认推送对象

矩阵缺口：若现网对「访客 / 无登录」是否注册 FCM 有单独策略，不阻塞本需求实现；服务端以 「是否存在有效 Token」 为最低条件，身份绑定规则由研发在技术 spec 与现网对齐后回写 §2.0 脚注（⚠️ 研发深化）。

2.1 典型场景

场景 A — 进程在后台，系统未强杀

  1. 服务端按 §5.2 频控下发一条 Data-only、high priority 消息。
  2. 客户端 onMessageReceived 被调用，执行 §5.1 轻量唤醒逻辑。
  3. 常驻通知保持或恢复为现网定义的有效展示态。

白话：第 1 步服务器发「隐形」消息；第 2 步 App 在后台被系统叫醒一下（能叫醒多少看机型）；第 3 步通知栏里那条 FlareFlow 常驻通知要还在、且状态对——QA 验的是用户眼里第 3 步，不是去背第 2 步函数名。

场景 B — 进程已被杀但未强制停止

  1. 同场景 A 步骤 1。
  2. 可能拉起应用进程并执行 Service；也可能不达（OEM / 电量策略）。
  3. 产品口径：尽力而为；用户下次冷启动仍应得到一致业务状态（依赖其它同步机制的不在本需求展开）。

场景 C — 用户强制停止

  1. FCM 可能长期无法投递至客户端。
  2. 不视为客户端 Bug；服务端仍须遵守频控，避免无效轰炸。

3. 核心动线（系统视角）

sequenceDiagram participant S as 服务端调度 participant F as FCM participant A as Android App participant FS as ForegroundService S->>S: 频控/合并/退避校验 S->>F: Data-only, high priority F->>A: 投递 Data 消息 A->>A: FirebaseMessagingService.onMessageReceived A->>FS: 轻量：确保常驻/startForeground 有效态 Note over A,FS: 不展示 FCM 通知 UI；不新增 App 内弹层

白话：从左到右就是——服务器 → 谷歌推送 → App 被叫醒 → App 去确认「常驻通知还在显示」；全程不要多弹一条用户能点开的广告推送，也不要在 App 里突然弹窗。

异常分支（文字）

  • 频控拦截：服务端在调度层丢弃或顺延发送，不产生客户端行为。
  • 客户端解析失败：丢弃 payload，不崩溃；打点 ⚠️ 研发定义事件名。
  • 执行超时风险：重逻辑不得阻塞 onMessageReceived；必要时入队 WorkManager（仍须轻量，见 §5.1）。

白话：App 被叫醒后几秒内就要做完小事；重活排队慢慢做，别把 App 卡死（闪退/无响应）。

5. 功能详细描述

5.1 🔴 FCM Data 接收与客户端唤醒处理

5.1.1 功能说明

在 Android 端接收 Data-only 且 high priority 的 FCM 消息，在不向用户展示本推送意图的前提下，轻量触发既有 常驻通知 / Foreground Service 逻辑，使前台能力在系统允许范围内尽快恢复。

白话（QA）：手机收到这条看不见的推送后，App 要做的事很简单：走线上一套已经写好的代码，保证「通知栏常驻那条 FlareFlow」还在；不要另起炉灶再搞一条通知，也不要弹 App 内提示。

5.1.2 触发条件

  • FCM SDK 回调 onMessageReceived 被调用，且消息为 纯 data（无 FCM notification 段或等价判定以研发与 FCM 文档为准）。

白话：只有「纯数据推送」这一种才会走进本需求；带系统通知样式的那种推送不算本需求（研发会按文档区分）。

5.1.3 入口位置

  • 无用户入口；系统级回调触发。

5.1.4 页面结构

  • 不适用（无新页面）。

5.1.5 UI 示意

  • 见 §4.2；本功能不得新增可见推送样式。

5.1.6 按钮交互

  • 不适用。

5.1.7 弹窗规则

  • 不适用；禁止因本消息新增 App 内弹窗。

5.1.8 状态清单

状态 描述 客户端行为
正常 payload 字段在 §5.1.11 约定范围内 执行唤醒逻辑
未知扩展字段 含未识别 key 忽略未知字段，不影响主流程
畸形 JSON / 空 body 无法解析 丢弃 + 打点，不崩溃
Service 已有效 常驻已处于现网定义有效态 禁止重复震动/重复创建多余通知 ID（与现网一致）
执行时间压力 系统对广播/服务有时限 只做轻量逻辑，必要时入队

5.1.9 异常处理

异常 处理
空内容 丢弃，不打用户可见提示
超长 payload 服务端应避免；客户端截断或拒绝解析，不崩溃
无网络 本消息不依赖回传网络即可完成本地唤醒逻辑
无权限 通知权限缺失时的行为与现网一致；本需求不新扩权限文案
并发 短时间内多条：幂等处理（见 §5.1.11）
格式不符 丢弃 + 打点

5.1.10 联动规则

  • 与常驻 ForegroundService（研发）：唤醒逻辑必须调用现网统一封装（例如 PlaybackForegroundController.ensureStarted() 类命名由研发决定），禁止复制第二套 startForeground。

白话（QA）：收到透传之后，必须出现的是「现网那一条」常驻通知栏效果——和以前代码里已经实现的那套完全一致；不允许为了本需求再写一套「第二条常驻」或两套互相打架的通知逻辑。用人话说就是：FCM 推送后，要按老规矩把常驻通知栏显示/维持好，不要重复造轮子。

  • 与埋点 / 账号：若需上报「唤醒成功」，不得阻塞主路径；失败可丢采样。

白话：如果要打日志/埋点，不能让用户感觉「卡了一下」或等很久。

5.1.11 数据规范（Data payload · 业务字段）

以下为 PM 级业务字段；JSON 具体嵌套层级（data 平铺 vs 嵌套）由研发按 FCM 限制实现，须保证与 iOS 无关字段不污染 iOS（本需求虽不做 iOS，仍建议结构清晰）。

字段名（建议） 英文 key 类型 必填 说明
协议版本 v int 是 从 1 开始；客户端 <1 或未知 丢弃
消息幂等键 id string (UUID) 是 去重窗口 ≥ 24h（与 §5.2 日上限一致即可）；同 id 重复投递不得重复执行昂贵副作用
发送时间戳 ts string / long 否 Unix 秒或毫秒，服务端填发；用于客户端日志与乱序检测
唤醒原因码 reason string 否 枚举建议：HEARTBEAT（默认）；扩展值须版本化，未知则忽略

校验规则

  • v 必须为整数 1。
  • id 长度 36（UUID）或研发认可的可比字符串；空则丢弃。

5.1.12 验收标准（可测试）

  1. 使用测试工具发送 仅 data、high priority 消息：系统通知栏不出现由 FCM notification 自动插入的新条目。
  2. 消息合法时：在测试机上 10 秒内（具体阈值由 QA 与机型集共同确认）可观察到常驻通知对应的前台服务处于现网有效态（判定方式：通知 ID 存在、adb 服务状态或产品认可的等价探针）。白话：发完透传后，10 秒内下拉通知栏，应仍能看到（或重新看到）那条 FlareFlow 常驻通知，且表现与发透传前线上一致；研发若用 adb 查后台服务，只是辅助手段，QA 以肉眼 + 与线上一致为主即可。
  3. 同一 id 在 24h 内重复发送 2 次：客户端 expensive 副作用（若定义）只发生 1 次；不 ANR。
  4. 畸形 body：不崩溃，有可查询日志或埋点。
  5. 不新增 App 内可见弹层或可点击营销推送。

5.2 🔴 服务端发送与频控策略

5.2.1 功能说明

服务端（或受权推送调度服务）向 Android 设备发送 FCM Data-only 消息，一律 priority=high，并实施 间隔 / 日上限 / 分钟合并 / 自适应退避，避免无效高频与平台风险。

5.2.2 触发条件

  • 调度策略允许（非静默时段若启用）、设备 Token 有效、未被风控拉黑。

5.2.3 入口位置

  • 内部调度 / 配置后台（具体产品名不锁死）。

5.2.4 页面结构

  • 不适用终端页面；若存在运营配置页，为 🟡 另案。

5.2.5 UI 示意

  • 不适用。

5.2.6 按钮交互

  • 不适用。

5.2.7 弹窗规则

  • 不适用。

5.2.8 状态清单

状态 描述
正常发送 通过 FCM API 返回成功受理
Token 失效 标记删除，停止对该 Token 发送直至重新注册
超频 本地调度层丢弃或顺延
静默时段 若启用：该时段内不发送

5.2.9 异常处理

异常 处理
FCM 429 / 配额类错误 退避重试；不得 busy-loop
部分设备失败 记录失败原因采样；不影响其它设备
配置错误（误设 low priority） 发布前校验：唤醒类消息必须为 high

5.2.10 联动规则

  • 与客户端幂等：同一 id 不重复构造；服务端日志可追溯。
  • 与「进程已活跃」检测（若实现）：若 App 在 T 分钟内已有前台会话或心跳回执，可将下次发送间隔拉长至 ≥ 45～60 分钟（默认值放远程配置，不锁死具体键名）。

5.2.11 数据规范（频控参数 · 远程可配）

参数 默认值（本 PRD 拍板） 说明
min_interval_ms ≥ 1_800_000（30 分钟） 同一设备相邻两条唤醒消息最小间隔
max_per_day 48 每设备每自然日最大条数
merge_window_sec 60 同一自然分钟内多次触发合并为 1 条
active_backoff_multiplier 1.5 ～ 2.0（范围） 若实现「已活跃检测」，在 min_interval 上乘以该系数

以上默认值须可通过远程配置热更新，无需发版即可调低频次（推荐能力）。

5.2.12 验收标准（可测试）

  1. 对单测试设备连续请求发送 3 条 / 1 分钟：服务端实际下发 ≤ 1 条。
  2. 对单测试设备在 24h 内统计：成功下发 ≤ 48 条。
  3. 任意下发消息的 FCM 优先级字段在监控或抽样日志中为 high。
  4. Token 失效后：调度不再向该 Token 发送（直至重新注册）。

5.3 🟡 观测与告警（摘要）

  • 指标建议：发送数、受理成功数、客户端到达埋点（若可）、onMessageReceived 执行耗时采样、解析失败率、Token 剔除率。
  • 告警：解析失败率突增、FCM 错误率突增（阈值由运维定义）。

6. 文案规范

6.1 对用户可见文案

  • 本需求不引入新的用户可读营销文案。
  • 常驻通知标题/副标题/渠道名称以现网 + 法务审核为准；若因唤醒逻辑需微调技术占位字符串，须走文案负责人确认。

6.2 对内部与日志

  • 日志使用英文 key（wake, id, v）与中文说明混合即可；禁止在 Logcat 打印完整 Token 或 PII。

7. 非功能性需求

类别 要求
合规与商店叙事 对外说明须与「短剧后台服务 / 续播同步」等真实用户价值一致；避免单独宣传「纯保活」话术（合规由法务/产品最终口径为准）。
可靠性 不承诺 100% 到达；须在产品说明或客服话术中可解释「省电 / 强制停止」影响。
性能 onMessageReceived 主路径耗时 P99 目标 < 2s（⚠️ 研发可调整，但须无 ANR）。
耗电 遵守 §5.2 默认频控；异常监控发现过量自动降级远程配置。
安全 payload 不得携带可执行指令或敏感密钥；若未来扩展签名校验，单独立项。
隐私 payload 默认不含用户身份明文；需要设备映射时使用服务端内部表。

8.2 前端影响详情（Android）

  • 调用既有 ForegroundService 启动/维持封装；禁止新建与现网常驻冲突的第二通道（除非产品明确要求双常驻，本 PRD 未要求）。

白话（QA）：研发会改「收推送的那块代码」，但用户能验收的仍是：透传来了之后，常驻通知栏那条 FlareFlow 按老规矩还在；不会出现两条常驻打架。

8.3 后端影响详情

  • 新增：面向内部调度的「唤醒消息发送」能力（可合并进现有 push 服务）。
  • 修改：设备 Token 生命周期管理、频控参数读取远程配置。
